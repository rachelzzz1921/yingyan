## 1 · 鹰眼不变量 —— MUST 对齐，**语义不可改、词不可换**

事后**行政飞检**：稽核员对**单份就诊材料包**做语义级三方交叉验证（诊断–操作–费用），对齐国家医保智能监控**两库（知识库 KB / 规则库）**与《医疗保障基金使用监督管理条例》**第 38 / 40 条**；**不是事中实时 pass/fail**，是事后排序与定性。

### 1.1 数据核心 Case → Finding
一个 Case = 一份就诊材料包（一次就诊，一个患者可多 Case）；一个 Case 产 0..N 个 Finding。

### 1.2 Finding 三态（核心 vocab，禁换词）
| 状态 | 含义 | 落库 |
|---|---|---|
| `疑点` | 三要素齐备、本包内可闭环、可直接对质 | ✅ |
| `线索` | 模式异常但本包内无法闭环（**必带 `needs_more[]`**） | ✅ |
| `不输出` | 依赖未证实假设 / 引不出原文 | ❌ 零记录终态 |

- **`status` 枚举只有 `"疑点" | "线索"`**；`不输出` 不落库。
- 官方对齐：`疑点 ↔ 明确违规`，`线索 ↔ 可疑/需申诉`。

### 1.3 Finding 三要素（`疑点`准入门禁，缺一降级）
① **原始证据定位** `evidence[]`（费用行号/病历页码 + 原文摘录）
② **政策条款原文** `policy[]`（KB 引用 ID + 条款原文，**禁凭记忆编造**）
③ **完整推理过程** `reasoning`

### 1.4 Finding schema（字段名不可改；本版新增字段见 §3、§4、§11，向后兼容）
```json
{
  "finding_id":"F20260612-001","rule_id":"B-201",
  "status":"疑点 | 线索",
  "violation_type":"...", "risk_level":"高|中-高|中|低",
  "amount_involved":1280.00,
  "evidence":[{"type":"费用行","loc":"...","text":"..."}],
  "policy":[{"ref":"KB-...","text":"...(原文)"}],
  "reasoning":"...","needs_more":[]
}
```

### 1.5 降级/撤销机制（状态机 transition guard）
- **CoVe**：维持 / 降级线索 / 撤销(=不输出)
- **控辩裁**：辩方指出要素缺失或有效反向证据 → 降级线索或撤销
- **compliance-gate**：证据<2 / 政策未核验 / 推理过短 → 疑点→线索
- **L3 规则**：设计上只出 `线索`（跨就诊、外部资质数据等）

### 1.6 shadow（叠加标记，非状态）
`shadow:true` 不改 status，展示完整证据链但**不计入** `suspected_count/suspected_amount`；统计：`active=!shadow`，`suspected=active∩疑点`，`clues=active∩线索`，`shadowed` 单独沉底桶。

### 1.7 规则治理三态（与 Finding 三态**勿混**）
`draft → shadow → active`（+ `deprecated`）。

### 1.8 现有导入/批量接口（复用，勿重造）
统一产物 `medical_record` JSON：必备 `case_meta + front_page + fee_list.items[]`；可选按槽合并入院/病程/医嘱/检验/专科等，关键事实带 `anchor`。
- `POST /api/ingest`：`structured` / `document`(jpg/png 视觉抽取；PDF→PP-Structure sidecar) / `connector`(Mock-HIS)
- `POST /api/intake/batch`：多文件 base64 → 自动分类到 **17 槽** → 注册 `uploaded`；支持 JSON/CSV/纯文本/PDF图像 + `merge` 增量
- `POST /api/audit/batch`：对已有 `case_id[]` 入队（**非 raw 导入**）

### 1.9 优先指数基线（v1，本版在 §6/§5 增强，不推翻）
Case 级、仅 active(!shadow)；**硬分层**：含≥1 疑点的案件 > 纯线索案件；层内 `api_score = 100 · (EC·AMT·SEV)^(1/3) · HistoryPrior · Breadth · Outlier`，全权重 config 化、可展开归因。体检模式只换展示口径，不改三态/分数。

## 2 · 本版产品纲领（一句话）

真实飞检里**非主观违规（政策没吃透）远多于主观骗保**；**自查自纠已是飞检前强制前置**。故"提高可用性"= ①贴合被法规固化的真实工作流 ②压低误报 ③一键举证 ④同一引擎服务"稽核"与"自查"两侧。下列增强全部服务这四点。

---

## 3 · 增强① 违规性质维度（主观骗保 / 非主观差错 / 待定）

**为何**：罚则按《条例》分档——第40条欺诈骗保(移交/处罚) vs 第38条一般违规(退回/整改)，处理路径不同；真实违规多为非主观，定性错会放大或缩小处罚。

**Finding 新增字段**：
```json
"violation_nature": "主观嫌疑 | 非主观差错 | 待定"
```
**判定（config 化映射表 `violation_type → 默认 nature`，可被证据覆盖）**：
- **主观嫌疑（≈40条）**：虚假住院/挂床、空刷套刷凭证、伪造变造病历处方、串换倒卖药品、无资质开展并收费。
- **非主观差错（≈38条，默认）**：重复收费、分解收费、超标准收费、串换收费、编码错误致高编高套（无意）。
- **升级规则**：非主观项若"同一问题反复出现 / 规模异常 / 历史已被指出仍大面积存在" → 升 `主观嫌疑`（可配阈值）。

**影响三处**：
1. 优先指数：`主观嫌疑` 进 SEV 加权（与 §1.9 的40条加权合并，去重）。
2. 处置建议：`主观嫌疑→建议移交/处罚`，`非主观差错→建议退回+整改`，随 Finding 输出。
3. 举证（§6）：性质写入认定材料，便于答辩与比例处罚。

## 4 · 增强② DRG/DIP 违规逻辑接入（跨就诊类天然落 L3=线索）

**为何**：DRG 时代隐蔽违规正是"诊断–操作–费用"三方交叉的强项；其中跨就诊/跨结算的，决定性证据不在单包内，恰好落你的 **L3=只出线索**。

**新增 `violation_type` 目录（接进规则库）**：高靠入组(高编高套)、分解住院、转嫁费用、低标入院、服务不足、推诿病人。

**检测规则示例（building AI 实现为规则；标注层级与态）**：
| 规则 | 逻辑 | 层/态 |
|---|---|---|
| 术式无匹配耗材 | 手术操作（如"78.5500x005 股骨钢板内固定术"）在结算明细无对应耗材（无钢板）→疑点，人工判高靠/转嫁 | 单包 / 疑点 |
| 呼吸机时长不符 | 首页填报呼吸机使用>96h，但明细诊疗项时长<96h | 单包 / 疑点 |
| 分解住院 | 同机构短期内再入院且进同一 MDC/ADRG（患者身份+多次出入院时间比对） | **跨就诊 / L3 线索** |
| 转嫁费用 | 入院前后3天门诊费用 + 住院DRG费用 > 该组支付标准；或术前检查/化疗药分解至门诊 | **跨结算 / L3 线索** |
| 低标入院 | 门诊可治病例收治住院、在院仅检验检查/简单治疗以查体为主 | 单包 / 疑点或线索 |
| 病历-首页一致性 | 主诊断/主手术编码 与 病历记录不一致（见 §11） | 单包 / 疑点 |

**违规金额（DRG 专用，单独计算）**：不可简单加费用行。规范算法 = **剔除违规因素后重新结算，比较两次基金支付差额**；依赖 DRG 分组器/支付标准。若分组/支付标准不可得 → 出 `线索` + `needs_more:["DRG重结算所需:分组器/支付标准"]`，**不得伪造金额**。

## 5 · 增强③ 特例单议 / 合理高倍率 = 一等"反向证据"（压误报）

**为何**：高倍率/住院时间长/费用高/新药耗新技术/复杂危重/MDT 病例可申报**特例单议**，是正当例外；已批准的大额离群案件**不是违规**。误报每降一档，稽核员对排序的信任高一档。

**Case 新增字段**：
```json
"special_case_review": "无 | 申报中 | 已批准 | 未批准"
```
**两处生效**：
1. **优先指数 Outlier 信号**：`special_case_review == 已批准` → **撤销金额离群加分**（该案件的大额是被许可的）。
2. **控辩裁**：`已批准` = 有效反向证据 → 命中"金额/资源消耗异常"类规则的 Finding **降级线索或不输出**（对齐"除外情形成立→不输出"）。

## 6 · 增强④ 一键「举证包 / 证据链导出」（把 anchor 变成"5分钟举证"）

**为何**：飞检质疑时须立即给完整证据链；现场核实要求≥2人、亮明身份、保存笔录/发票清单/病历等。鹰眼 evidence+anchor 是底座，缺导出。

**新增端点**：`POST /api/evidence-package`（入参 `finding_id` 或 `case_id`），生成结构化举证包并渲染 PDF。

**举证包内容（每个 Finding）**：
- 案件标识：`case_meta`（可脱敏）、就诊信息、违规性质、处置建议。
- **三要素全展开**：证据定位+原文摘录、KB 条款**原文+引用ID**、推理过程。
- 违规金额 + **算法说明**（项目付费=对应费用；DRG=剔除重结算差额）。
- 审计链：CoVe / 控辩裁 / 集体决策结论 + 时间戳（见 §10）。
- 输出格式：飞检就绪 PDF + 可填询问笔录/违规点认定的结构化 JSON。

## 7 · 增强⑤ 自动产出官方「违规点认定 + 违规费用统计表」

**为何**：真实飞检流程被法规固化——人工核实→违规点认定材料→按违规点统计数量与金额→违规费用统计表→汇总核对→反馈申辩→集体决策。能否"插进"该流程直接决定落地摩擦。

**新增端点**：`GET /api/report/violation-summary?case_ids=&group_by=violation_type|nature|dept`

**产出三张表（Excel + PDF 导出）**：
1. **违规点认定材料**：按 `violation_type` 分组，每点含：认定依据（KB 条款）、涉及 Case/Finding 列表、证据引用、数量、金额。
2. **违规费用统计表**：违规点 × 数量 × 金额 × 合计；可再按性质（主观/非主观）、科室、医生分列。
3. **汇总表**：总违规点数、总金额、按性质/领域分布。

要求：数字与 Findings 完全一致；shadow 不计入；体检模式只改表头口径（飞检暴露金额/风险点数）。

## 8 · 增强⑥ 历史画像 + 9大重点领域靶点 + DIP 辅助目录维度

**为何**：飞检靶向明确（肿瘤/麻醉/重症 + 心血管/骨科/血透/康复/影像/检验 = 9大方向；重复收费占违规约36%）；DIP 侧有现成风险维度可对齐。

**实现**：
- **Case 风险标签** `risk_tags[]`：命中9大重点领域则打标；并对齐 **DIP 违规监管辅助目录**5维：病案质量指数、二次入院评分、低标入院评分、超长住院评分、死亡风险评分（有数据则计算，无则留空标 `unavailable`）。
- **HistoryPrior 画像**（喂 §1.9 优先指数）：按 **科室 / 医生 / 病种(DRG组)** 维度的历史命中率与金额；命中重点领域的案件在优先指数加权（config）。
- 预置可维护的**重点领域问题清单**作为规则标签来源。

## 9 · 增强⑦ 体检模式对齐「自查自纠问题清单」，医院自查做成一等场景

**为何**：自查自纠已是飞检前强制动作，医院须对照问题清单先查先改；根因多为"政策不理解"，工具应兼具教育性。

**实现**：
- 新增**自查自纠问题清单**管理（`/api/checklist`，可加载国家/本地清单，映射到规则集 profile）。
- **双边模式**：`稽核模式`（第三方/医保飞检）与 `体检模式`（医院自查）共用同一引擎与三态；体检模式用飞检同款规则在本院数据上预演。
- 体检模式每条疑点**附 KB 条款原文**做教育说明；输出口径切"风险点/飞检暴露金额"（§1.9 不变量：不改三态/分数）。

## 10 · 横切Ⅰ 可解释 + 留痕对齐法定流程

**为何**：《飞检办法》要求"如实记录、认真审核、充分研判、集体决策"，争议须畅通申辩；这既是合规刚需，也是第三方工具的护城河。

**AuditRecord 增强**：
```json
"collective_decision": { "participants":[...], "conclusion":"...", "decided_at":"..." },
"defense": { "submitted":"...", "counter_evidence":[...], "disposition":"维持|降级|撤销" },
"timeline": [ {"event":"...", "at":"...", "actor":"..."} ],   // 如实记录
"appeal_channel": "..."   // 申诉渠道
```
全程留痕：谁查看/稽核/改 config/集体决策，均可追溯、可复现。

## 11 · 横切Ⅱ 数据源对齐真实稽核

**为何**：真实筛查依赖结算清单 + 病案首页(DRG填报) + DRG/DIP分组 + 药品追溯码(2025-07 起强制扫码结算) + 住院运行指标；**病历与清单/首页是否一致**是审核重点。

**17 槽中显式一等槽位**（命名对齐）：`settlement_list`(医保结算清单)、`front_page`(病案首页)、`drg_grouping`(DRG/DIP 分组数据)、`trace_code`(药品追溯码)、`inpatient_metrics`(平均住院日/次均费用/时间消耗指数/费用消耗指数)。

**新增标准三方交叉检查**：`病历–首页一致性`（主诊断/主手术/编码 vs 病历记录），不一致→疑点（高编高套审核核心）。

## 16 · 验收标准（针对本版增强）

1. **性质维度**：虚假住院类默认判 `主观嫌疑→建议移交`，重复收费类默认 `非主观差错→建议退回整改`；同一非主观问题反复出现触发升级为主观（按 config 阈值）。
2. **DRG**：术式无匹配耗材案例检出疑点；分解住院/转嫁费用案例落 **L3=线索且带 needs_more**；DRG 金额在缺分组器时不出数字、只出线索。
3. **特例单议**：已批准的高倍率大额案件，优先指数离群加分被撤销；命中异常类规则的 Finding 经控辩裁降级线索或不输出。
4. **举证包**：任一 Finding 一键导出，含三要素全展开 + KB 条款原文 + 金额算法 + 审计链时间戳；脱敏开启时无明文 PII。
5. **官方产物**：按 violation_type 生成认定材料 + 费用统计表，数字与 Findings 一致、shadow 不计入；可切 group_by=nature/dept。
6. **画像/靶点**：命中9大领域案件优先指数加权生效；科室/医生/病种命中率随历史更新；DIP 辅助目录维度无数据时标 unavailable 不报错。
7. **双边模式**：同一案件在稽核模式与体检模式三态/分数一致，仅展示口径与教育说明不同；自查清单可加载并逐条对照。
8. **法定留痕**：集体决策/申辩/反向证据/时间线完整可追溯；优先指数 config 变更入审计日志。

## 17 · 硬约束（再强调）

- 第 1 节 vocab/schema **照搬**：三态只 `疑点/线索`、`不输出`零记录、三要素门禁、shadow 不计分、规则治理三态勿混、L3 只出线索。
- 复用现有 `/api/ingest`、`/api/intake/batch`、`/api/audit/batch`，**勿重造导入**。
- 新增字段向后兼容；不得伪造金额/证据（DRG 无分组器→线索+needs_more；链下事实→带 `verified_by`，禁写 `cmd`）。
- 优先指数权重、性质映射表、重点领域清单、特例单议抑制全 **config 化、可解释、可审计**。这是**事后排序与定性工具，不是事中 pass/fail**。
- 不确定的实现细节可合理补全并在 README 标"假设"；**对齐第 1 节的部分不得改动**。

## 附录 A · 金标准评测集（回归锚点 / few-shot）

**用途**：固定输入 → 期望输出，作回归测试与 few-shot 锚点。引擎跑下列就诊材料包，产出的 `status / violation_nature / 是否进 api_score / needs_more / 离群抑制` 必须与"期望"一致。**评测只验行为一致性，不得借此改动三态语义或 api_score 公式定义。** 已覆盖：主观/非主观、疑点/线索/不输出、跨就诊 L3、特例单议抑制、shadow 不计分、DRG 缺分组器不出金额。
