## 给 Claude Code 的话(先读这段)

你现在处在一个**能联网、能用真实 API 调多个模型**的环境里。前序两轮工作(主文档 v1.0 把 7 个 prompt 迭代到 v3,v2.0 红队把它们推到 v6)是在一个**无网、只能在浏览器里单模型冒烟测**的容器里做的——所以那两轮的"测试"要么是对抗推演,要么是单次手动跑,**都不是真实回归打分**。

**你的任务:把它落到真实评测。** 对 v6 prompt 跑真实、多模型、多次运行的回归,找出真实失败,据失败把能在 prompt 层修的修到 v7,不能在 prompt 层修的诚实记下来。

**最重要的一条纪律**(这套东西的灵魂):**绝不编造分数**。任何你没真用模型跑出来的数字都不许出现在报告里。测不了的,就写"测不了,原因是 X"。这一条优先于"看起来完成了任务"。

## 3. 评测方法(硬要求)

1. **真调模型,零模拟。** 不允许任何 mock / 写死的"示例输出"冒充结果。
2. **多次运行,报通过率与方差,不要单次判定。** LLM 是随机的;每个用例每模型跑 **N≥5** 次,记 pass rate(如 4/5)和波动。单次绿不算稳。
3. **断言必须机器可判**,不许"看起来对"。种子断言见 v2.0 + 验证台,至少覆盖:
   - 全部:输出可解析为 JSON(`is-json`)。
   - P1:`injection_suspects` 非空 **且** `fee_lines` 非空(注入被拦但真实数据照抽);篡改型注入用例额外校验金额未被施加变换。
   - P2:三态符合预期;政策为空→线索且不编条款;**政策不对口→不得疑点**(`policy_status≠适用`)。
   - P3:无根据候选→`initial_status=不成立`;有效疑点→有 `cove_checks` 且有钢人 `anticipated_defense`。
   - P4:铁证→`concede_violation=true`;有除外→建议非"维持疑点"且 `rebuttals[].relevance` 非空。
   - P5:见第 4 点。
   - P6:同行三规则→合 1 条、金额去重 = 单行额、`omitted_lines` 空;不同行→不合并、金额各计。
   - P7:单次具体理由未达阈值→"待观察";扎堆单一稽核员→`review_anomaly=true` 且沉淀含"审计"。
4. **P5 真位置去偏(只有真实多模型环境能做,务必做):**
   - 同一案例把 A/B **交换位置真跑两遍**,`verdict` 必须一致;不一致 = 位置偏差,判失败。
   - **裁判模型与辩手模型必须异源**(占位 B),以真测自我偏好。
   - 记录全套 P5 案例的**位置一致率**。注意:这要跨两次调用比对,promptfoo 单条测难表达,**写个独立 node runner**(§5)。
5. **继续红队(关键,别只复现旧案例):** 不要止步于 v2.0 的 C 案例。**主动构造 v2.0 没预见的新对抗输入**去打 v6——注入变体(如把篡改指令藏进合法字段值里)、边界(空材料/超大材料/单位混用)、歧义(同名不同物)、对抗(自相矛盾材料、诱导跳步)。目标是**找到新的真实失败**,每发现一个就进 `cases/` 并尝试在 v7 修。
6. **温度策略:** 裁判(P5)用低温(求一致性);其余可在低温跑准确性的同时,另跑一组略高温**测鲁棒性**(prompt 在抖动下是否仍守纪律)。报告注明温度。

## 4. 迭代规程(失败驱动)

- **只在出现真实失败时改 prompt。** 没失败的就标"v6 已稳",不为改而改。
- 每次改动写进 `CHANGELOG.md`:**改了哪条指令、根因、对应哪个失败用例、改后该用例是否转绿、全量重跑有无回归。** 风格对齐 v1/v2 的"改了什么/为什么"。
- **改完必须全量重跑**(防止修一个、退三个)。
- 迭代到收敛或预算上限。v7 不强求每个都动——**有的 prompt 经真实测试就是稳的,如实说**。
- **不要在 prompt 层硬解、要记进 `OPEN_ISSUES.md` 的(它们是架构/离线问题,不是文案问题):**
  - P5 真位置去偏的**最终保证**(调用层把 A/B 跑两遍取一致、不一致降线索)与**异源裁判编排**——你能**测**它,但"修"它在调用编排层,不在 prompt 文本。
  - 传入的规则定义本身畸形(应由 KB 准入校验)。
  - 准标识 k-匿名度量(离线统计)。
  - 跨超长材料的实体合并(调用层分块)。
  - 这些在报告里区分清楚:哪些是"prompt 改好了",哪些是"架构待办"。

## 5. 技术提示(省你踩坑)

- **promptfoo** 适合 P1–P4、P6、P7 的"单次调用 + 断言"回归。`defaultTest` 挂 `is-json`;每条用例的字段校验用 `type: javascript` 写小断言(可直接移植验证台里的 `fn`);多次运行用 `repeat`。把 v1.0 附录 C 的骨架扩成全量。
- **P5 用独立脚本,别硬塞 promptfoo。** 位置一致性要跨两次调用比对,且裁判要换模型——写个 `p5_swap_runner`:对每个案例,(a) 调辩手模型/或直接用预置控辩论点,(b) 用**裁判模型**跑 `(A,B)`,(c) 再跑 `(B,A)`,(d) 归一化 `verdict` 比对一致性,(e) 汇总位置一致率。归一化把"维持疑点→疑点""降为线索→线索"等折叠。
- **JSON 提取**要稳:去 ```json 围栏、取首个 `{` 到末个 `}` 再 parse,失败计入"JSON 不合规"而非崩溃。
- **省钱:** 先用 1 个 prompt(建议 P1)端到端打通整条 pipeline(提取→配置→断言→多次跑→报告),确认无误再铺开 7 个;能缓存的缓存。

## 7. 贯穿始终的纪律

- **诚实第一。** 没真跑出来的数字一律不写。这比"显得做完了"重要。
- **证据纪律。** 断言机器可判,失败有据可查,别用主观判断当结论。
- **简洁。** 报告讲清:失败在哪、为什么、改了什么、改后如何;不灌水。
- **模糊处:** 先做合理假设并在报告里写明,只有"会显著改变方向"的事才停下来问用户(例如预算明显不够、模型选型需要拍板)。

## 8. 建议起步顺序

1. 读 v2.0 md,提取 7 个 v6 全文到 `/prompts/`。
2. 拿 **P1** 端到端打通 pipeline(配置 + 断言 + N 次跑 + 报告片段),验证无误。
3. 铺开其余 6 个;为 P5 写 swap runner。
4. 跑 **baseline**(全部用例、多次、多模型),出 `results/baseline_<date>`。
5. 分析真实失败 + 继续红队补新案例。
6. 失败驱动迭代 → v7,每步记 CHANGELOG、全量重跑防回归。
7. 出最终 `report.md` + `OPEN_ISSUES.md`,逐条对照 §6 验收。
