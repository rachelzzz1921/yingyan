| 项 | 内容 |

| **核心论题** | **人少事多,谁能解决谁就是胜者(江苏省医保局专家原话)。** 全国医保监管员仅 **8600 人**盯 **13 亿参保人 / 28.99 万家机构**,人均监管量是**美国 4 倍**;**鹰眼 = 打破"人少事多"瓶颈的 AI 人力倍增器**——替人扛掉海量非结构化案卷初筛(单案 40 分钟→90 秒),让稀缺专业人力只花在真争议、真违规上。 |
| **一句话** | 人少事多场景下的 AI 人力倍增器：读懂非结构化案卷，90 秒输出可对质的**三要素证据链** |
| **铁律** | **引不出原文的疑点，不输出**;初筛以高精确率为先(误报先降级),漏判由优先通路全量扫描+人工复核+国家大数据兜底 |
| **地位** | 站在官方两库**已公开发布批次**肩上的**语义增强层**；国家系统的**取证放大镜**，不是替代品 |

## 二、国家系统很强，为什么我们仍有场景

### 2.0 监管侧价值：评委以监管侧为主，先讲透（对外第一叙事）

> 一句话:**全国 8600 名监管员盯 13 亿参保人 / 28.99 万家机构(人均是美国 4 倍),飞检只查得过来 500 家;鹰眼就是替这群最缺人、最怕担责的飞检/审核员扛量的"取证放大镜"。**

鹰眼对监管侧的具体用处（每条 = 替哪个人力瓶颈）：

| 监管侧用处 | 替掉的人力活 | 产品支撑 |
|---|---|---|
| **① 飞检案卷取证** | 资深专家逐本翻病案+文书找证据（单案 40 分钟） | `/api/audit` 90 秒出三要素证据链、可点击回链原文、举证包 |
| **② 优先排序·先查高风险** | 面对上万案卷不知先查谁 | 稽核优先通路 `api_score` 排序，把有限飞检人力投到最高风险案卷 |
| **③ 事中存疑单语义复核** | 审核员逐单调病历核「费用-医嘱-诊断」是否一致 | 深读非结构化案卷做语义复核，只接「需核病历」的存疑单 |
| **④ 机构画像·抽样指导** | 飞检前靠经验猜哪个科室/规则高风险 | 单案升维机构画像：定位高频违规规则/科室，指导现场抽样 |
| **⑤ 误报前置过滤·省复核人力** | 疑点越多、误报越高，逐条人工排雷 | 「引不出原文不输出」+ 申诉预演 → 进入人工的都是「扛过一轮申诉」的，直击第一痛点（误报，不是漏判） |
| **⑥ 可审计覆盖度声明** | 说不清「查了什么、没漏什么」 | `coverageManifest` 把漏检率变成可签字声明——监管验收的命门 |
| **⑦ 源头减负（间接）** | 8600 人扛不动 28.99 万家、只能抽查 500 家 | 鹰眼赋能院端把历史案卷**自查干净、主动退回**（「自查从宽被查从严」）→ 流到飞检/审核的问题变少，**从源头替监管侧卸载工作量**（官方「关口前移/源头治本」） |

**对监管侧金句**：*"国家'一张网'已经把'谁先被查'做到了毫秒级；鹰眼补的是下一段——飞检进场后，把那一整本非结构化案卷读成能直接写进检查结论、经得起申诉复查的证据。我们不替代稽核员、不做 AI 处罚，只做替人扛量的初筛与举证，和官方'AI 提供线索支持'的口径完全一致。"*

**法定落地通道**：《飞行检查管理暂行办法》（国家医保局令第 6 号）允许**聘请 IT 服务机构协助飞检**——鹰眼有现成合规入口。

### 2.1 对方质疑（常见）

> 「国家医保已有大数据库、智能监管两库，飞检早就加速了，你们没有使用场景。」

### 2.2 哪些说法是对的

| 事实 | 说明 |
|---|---|
| 国家已建**事前、事中、事后**智能监管体系 | 两库（规则库+知识库）是核心；各地智能监管子系统对接结算数据 |
| 官方全量 = **88 类规则 / 24.7 万知识点**（《2025 年版》），向社会**已公开约 16 批**（截至 2026-06，第十五批 6/16、第十六批 6/22） | 机构可嵌入 HIS 或对接事前提醒模块；批次滚动、引用前核 nhsa.gov.cn 最新 |
| **2026 年底**力争定点机构事前提醒接入率 **70%+** | 见国家医保局 2026 年基金监管工作部署 |
| 飞检前常用**大数据筛疑点**，现场再核病历与收费 | 「四不两直」+ 交叉互查已成常态 |
| **2025 年全国追回约 342 亿元** | 说明监管体系在运转，也说明存量违规规模仍大 |

**结论**：对方说「国家有大数据、能加速飞检」——**基本属实**。不应硬杠。

### 2.3 哪些结论过头了

「因此不需要鹰眼」**逻辑跳跃**。国家体系强在：

- **结构化结算数据**的大规模筛查  
- **可编码规则**（限性别、限疗程、重复收费等）  
- **事前提醒、事中审核**  

但以下仍是留白或政策中的**探索方向**：

| 留白 | 说明 |
|---|---|
| **非结构化案卷** | 飞检现场：扫描件、PDF、多系统拼包的病历+病程+医嘱+检验 |
| **语义级交叉验证** | 「费用—诊断—医嘱」三方一致；如靶向药与分子病理/基因检测证据是否匹配 |
| **案卷级取证** | 从「疑点 flag」到「可写进报告、可拿去对质」的三要素 |
| **全病历 AI 判读** | 《监督检查五年行动计划（2026—2030）》仍写「探索人工智能+病例判读」等 |

### 2.4 流程分工（一张图）

```
国家大数据 / 两库          鹰眼
─────────────────          ─────────────────
锁机构 · 锁科室 ·          单份案卷深度初筛
锁结算疑点                 三方交叉 + 三要素
         │                        │
         └──────────┬─────────────┘
                    ▼
              飞检组进场 · 对质 · 处罚/整改
```

- **国家系统加速的是**：谁该被查、哪些结算项目可疑（A→B）  
- **鹰眼做的是**：单份案卷怎么在 90 秒内初筛、怎么写出证据链（C→D）  
- **关系**：上下游互补，不是重复建设  

官方亦**鼓励**机构主动对接两库、把两库置于自建信息化系统中做自动提醒+自查自纠——这是在给院端合规工具留口子。**我们不造规则、只接已公开批次、同源可对接**；官方对 AI 的定位是"提供线索支持"，与我们初筛层同构。

## 三、差异化竞争矩阵

| 维度 | 国家医保智能监管（两库+大数据） | 鹰眼 |
|---|---|---|
| **数据主粮** | 结算清单、医保编码、平台对接 | 案卷包：病历+医嘱+检验+费用（含扫描件） |
| **主场景** | 事前提醒、事中审核、事后**批量筛疑点** | **事后**飞检/专项/案卷复核的**单案深度初筛** |
| **规则形态** | 全国两库+省平台本地化；偏可编码规则 | 两库之上做**语义规则**（指征、除外、病程交叉） |
| **输出形态** | 疑点线索、拒付、排名 | **三要素证据链**，可对质、可附报告 |
| **误报哲学** | 机构申诉、复核流程 | **高精确率优先**(误报先降级)+ 控辩裁 + 漏判兜底 |
| **争议费用申诉** | 常统一打「可疑」或依赖人工复核 | **申诉机制预演**对质，自动降级/补材料清单 |
| **院端价值** | 开单时拦截新违规 | **自查自纠**、飞检前模拟、举证包/清单 |
| **关系** | 底座与规则来源 | **增强层** |

### 与院内 HIS「已接两库」的回应

> 事前提醒解决的是**当下这笔**别错；自查要扫的是**过去两年 12 个科室**的历史案卷。HIS 两库一般不会自动做「奥希替尼和分子病理报告」这种跨文档语义推理，也不会给院办一份带条款引用的自查清单。

## 四、双主线产品架构

### 主线 ① 单案卷稽核工作台

- **入口**：`index.html` · `POST /api/audit`  
- **流程**：导入 → 事实层 → 路由 → 判定 → 门禁 → 申诉预演 → 导出

- **体检模式** `?mode=exam`：院端措辞「风险点/整改」，引擎相同  
- 治理：三审三验、shadow/boundary 分桶、eval_draft（iter-36）

## 五、医院自查自纠：全流程与有效环节

### 5.1 国家规定的年度链条

```
准备期（通常 1–3 月）
  国家/省下发问题清单（12 领域本地化）
  → 院方调数据（结算+病历+费用）
  → 对照清单排查

整改期
  → 科室整改（停违规+补病历）
  → 追回/退回违规基金
  → 写自查报告，报当地医保局

自查报送之后
  → 医保部门抽查复查
  → 「四不两直」飞检（常自 4 月起）
  → 宽严相济：认真自查从轻；敷衍隐瞒从重、曝光、支付资格记分
```

**政策要点（院端自查 = 政策强制 + 官方鼓励，不是我们创造的需求）**：

- **强制**：国家医保局自查自纠 2025 覆盖 9 领域 → **2026 扩 12 领域**，回溯 2 年，每年铺开；不做就挨罚（心血管、骨科、肿瘤、麻醉、重症等）  
- **鼓励 + 从宽激励**：官方明文「**自查从宽、被查从严**」——期限前主动退回从轻/免罚；官方多份文件**鼓励定点机构主动对接两库、自动提醒+自查自纠**，把违规消除在萌芽  
- **与官方战略同向**：《监督检查五年行动计划（2026—2030）》三大举措含「**源头治本**」、监管「**关口前移**」——院端自查正是官方要推的源头治理，**卖院端自查 = 帮官方做它要做的事**  
- 自查自纠是机构**自我管理主体责任**的具体体现；飞检核查**自查质量**，不是报完就结束  
- **付费驱动是恐惧不是合规理想**：一次飞检罚款百万至千万，工具费是零头；ROI 锚定**「避免一次飞检损失」**，不是「提效」  

### 5.2 「查了、纠了」在哪些环节真有效？

监管认的是**可核验产出**，不是「跑了一遍系统」：

| 环节 | 医院做什么 | 为什么有效 |
|---|---|---|
| **排查有覆盖** | 按 12 领域清单扫规定区间全流程 | 飞检查是否走过场 |
| **问题有台账** | 科室、项目、金额、规则依据、责任人 | 抽查要「问题见底」 |
| **资金有退回** | **主动退回**违法违规使用的医保基金 | **宽严相济**核心：主动退回可从轻 |
| **行为有整改** | 改流程、改系统、培训 | 飞检突击看的是**日常**是否还犯 |
| **报告有报送** | 自查报告报医保部门 | 进监管档案；关联记分、信用 |
| **与事前衔接** | HIS 嵌入两库 | 减少**新发生**违规；自查扫**历史存量** |

**无效做法**：只填表不调病历；只改金额不退款；只查结算不做「费用—病程—医嘱」交叉；报告写「无问题」但大数据一核就有。

### 5.3 「自查自纠之后」还有哪些场景？

| 场景 | 说明 |
|---|---|
| **医保抽查复查** | 核报告 vs 数据 vs 退回记录是否一致 |
| **四不两直飞检** | 常带大数据疑点进场；自查已退回的通常按主动合规处理 |
| **支付资格记分、信用** | 自查记录是「尽到自我管理责任」的证明材料之一 |
| **年内专项整治** | 肿瘤/骨科/DRG 等轮次；防复发+新领域 |
| **飞检后申辩** | 争议项需说明依据；自查阶段若有证据链+整改记录，申辩成本远低于临时翻病历 |

**要点**：「自查报送之后」的主场景是**被验证、被对质**——不是闲置期，是监管验收期。

## 六、鹰眼在自查链条中的插入点

鹰眼贯穿 **查 → 纠 → 证 → 复检**，不只「纠完之后用一次」：

```
对照国家清单
    ↓
批量导入历史案卷 → api_score 排序（先查高风险）
    ↓
单案三要素（体检模式：风险点+依据）
    ↓
导出自查清单 / 举证包 → 科室整改+退款
    ↓
整改后复跑 → 飞检前模拟 / 申辩材料
```

| 阶段 | 院端痛点 | 鹰眼能力 | vs 两库事前提醒 |
|---|---|---|---|
| **查：清单太大** | 12 领域×两年数据，人工翻不完 | 优先队列，先跑高分案卷 | 两库管「当下这笔」；鹰眼管「历史哪份该深查」 |
| **查：语义类** | 靶向药指征、分解住院、病程与收费不符 | 语义规则+三要素（如 T-201） | 字段规则扫不出「全案卷未见分子病理/基因检测证据、病程亦无外院已测记载」 |
| **纠：不知改哪** | 科室理解政策不一致 | 体检模式：风险点+整改建议措辞 | `?mode=exam` |
| **纠：留痕不够** | 怕「说了改了没证据」 | `/api/export/checklist` | 可归档进自查台账 |
| **报：材料凑不齐** | 报告要附明细、金额、依据 | 举证包、违规统计表 | 减少飞检前加班 |
| **纠后：复检** | 改完不知是否还有漏 | 同案卷复跑；干净件零误报逻辑 | 模拟飞检「正确不报」 |
| **飞检后：申辩** | 争议项要能回链原文 | 三要素+控辩裁留痕 | 院端内部预审 |

**一句话**：

> 国家两库帮医院**少犯新错**；鹰眼帮医院**把旧账查清、查清能写进报告、改完能证明**。

**🔗 关键连接：院端自查做好 = 直接替监管侧从源头减负（所以院端产品对监管侧评委也是价值）**

> 监管侧「人少事多」的根因是 8600 人扛不动 28.99 万家机构。若鹰眼帮医院把历史案卷自查干净、主动退回、常态化合规，**流到事后飞检/审核的问题就少了**——这正是官方「关口前移、源头治本」的逻辑。**鹰眼两侧同一根引擎：直接给监管侧当飞检取证放大镜，间接靠院端自查替监管侧卸载工作量。**

### 哪些「纠」最有效？（对医院讲价值）

1. **主动退回基金**（最直接，宽严相济）  
2. **高频科室先纠**（肿瘤、骨科、麻醉等飞检重点）  
3. **大额+证据闭环的问题先纠**（与 api_score 一致）  
4. **系统性问题**改信息系统（重复收费、超标准、串换）  
5. **语义类问题**补病历、补检验、补指征  
6. **留下完整台账**（清单+金额+条款+整改日期）

## 七、控辩裁：申诉机制预演（核心亮点）

> ⚠ **没有「灰色地带」这个说法**（江苏省医保局专家原话）。正确机制是**申诉**：事中审核被退回可申诉、事后飞检判定后给申诉机会——医院基于**药理学/医理学**申诉，**医保专家团队评定打分**（行/部分行/不行），再正式回复。**可申诉的是医理类争议费用；法定申诉走六环、有 10 个工作日死线**。鹰眼的「控辩裁」本质 = 把这套申诉对质**预演一遍**，前置到初筛。

### 7.1 为什么这是独立卖点？

国家两库和 HIS 事前提醒擅长**黑白分明**的事：重复收费、超疗程、限性别用药。

真正吃人工、吃经验、吃扯皮的，是**医理类争议费用**（可申诉的部分）：

| 类型 | 例子 | 人工累在哪 |
|---|---|---|
| **证据在包外** | T-205：关键看前次血常规，不在本案材料里 | 要想「还缺什么」；医院辩「你没调全」 |
| **临床合理 vs 医保限定** | 家属要求用药、缺药院外购、护理等级 | 两边各有一套说法，要逐条对 |
| **政策留缓冲** | 医疗合理类默认「可疑/线索」、观察区、放化疗豁免 | 不能机械报「疑点」 |

**申诉 Agent 本质是误报过滤器** — 这是「高精确率优先、误报先降级」的工程实现，不是装饰;漏判由优先通路全量扫描与人工复核兜底。**把对抗式复核工程化用在案卷初筛**——不是「独创」，多智能体非医保独门，门槛在「用在案卷初筛+三要素裁定」的工程化。

### 7.2 多 Agent 在替人干什么？

现场真实流程：

```
控（稽核主张） → 辩（机构申诉） → 裁（定性 + 要不要补材料）
```

| 能力 | 省下的力气 |
|---|---|
| **辩方主动挑刺** | 不用自己「替医院想理由」 |
| **控方二次回应** | 先挡掉「家属要求」「外院已测」等无证据申诉（T-201） |
| **裁判按三要素裁定** | 缺证据 → 自动**降级线索**或撤销 |
| **附调阅清单** | T-205：明确去调「前次逐日血常规」 |
| **留痕写回** | `review_feedback` + 可选 `eval_draft` |

**机制要点**（`debate.js`）：

1. 只对**高风险 L2 疑点**和**争议线索**启动；**L1 硬证据不辩**（省成本）  
2. 辩论轮数上限 **2 轮**  
3. 裁判标准 = **三要素门禁**：辩方指出要素缺失或有效反向证据 → 降级/撤销  
4. 真· LLM 路径：**位置交换**裁两次，防模型偏见（P5 裁判）  

**对外金句**：

> **规则引擎负责「像不像」；控辩裁负责「敢不敢拿去对质」。**

### 7.3 与常规智能审核的差异

| | 两库 / 常规审核 | 鹰眼控辩裁 |
|---|---|---|
| 输出 | 违规/可疑 flag | flag + **正反论证 + 裁定理由** |
| 误报处理 | 事后申诉复核 | **事前模拟申诉**，能降级先降级 |
| 争议费用 | 常统一打「可疑」 | 显式区分疑点/线索/不输出 |
| 院端自查 | 难模拟飞检怎么怼 | 体检模式跑辩方，**飞检前预审** |
| 可解释性 | 「命中规则 X」 | 「辩方说了什么、为何驳回、裁凭什么」 |

### 7.4 院端 / 监管侧各一句话

**院端**：

> 不用等飞检组进场才知道会被怎么问；系统先替飞检组「吵一遍」，该补材料、该退款的，自查阶段处理掉。

**监管侧**：

> 拿到的不是一摞告警，而是**经过一轮对质筛选**的案卷：该硬的硬、该降级的降级、该补调的列清单——人力花在真争议上。

### 7.5 诚实边界

- 控辩裁**替代不了**临床专家终审（肿瘤指征、DRG 边界等）  
- 医疗合理类默认偏**线索**，与局方口径对齐  
- Demo 部分规则有**预置脚本**；真·路径：`POST /api/debate` + P5 裁判  
- 价值主张：**减少无效争议和误报升级**，不是「零人工」

## 九、对外话术与 Q&A

### 9.1 回应「国家已有大数据库」

> 您说的对，国家两库和大数据已经把「谁先被查」做得很快了，我们也不做实时结算拦截。我们做的是另一段：飞检和自查时，一份案卷要把费用、病程、医嘱对上，资深老师还得看很久；国家局文件里也还在试点「AI+病例判读」。鹰眼是把两库规则接到**全病历语义**上，输出能写进自查报告和申辩材料的**三要素**。和国家系统是**上下游**，不是重复建设。

### 9.2 回应「HIS 已接两库」

> 事前提醒解决**当下这笔**；自查要扫**过去两年 12 科室**历史案卷。HIS 一般不会自动做跨文档语义推理，也不会给院办带条款引用的自查清单。

### 9.3 三大卖点（建议并列）

1. **三要素证据链** — 引不出原文不输出；可对质  
2. **双主线** — 单案深度（工作台）+ 批量排序（优先通路）  
3. **控辩裁申诉预演** — 规则引擎判「像不像」，多 Agent 判「敢不敢拿去对质」  

### 9.4 评委 Q&A 速查
