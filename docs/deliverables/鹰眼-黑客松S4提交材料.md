## #7 团队优势(核心)

**一句话:我们不做"又一个医保审核工具",而是站在国家两库肩上、专为飞检取证与院端自查打造的非结构化案卷语义取证层——懂业务、敢认边界、工程纵深扎实。**

> **核心论题(一切价值的引擎)**:现在的问题是**人少事多,谁能解决这个问题,谁就是胜者**。全国医保监管员仅 **8600 人**,要盯 **13 亿参保人 / 28.99 万家机构**,人均监管量是**美国 4 倍**;单案人工初筛约 **40 分钟**。**鹰眼 = 打破"人少事多"瓶颈的 AI 人力倍增器**——不替代人,替人扛掉海量非结构化案卷初筛(40 分钟→90 秒),让稀缺专业人力只花在真争议、真违规上。

### A. 团队层面

- **真懂医保**:队长有券商医疗行研经历、北大年庆功博士全程指导,规则与知识库基于真实政策与临床逻辑,不是对着文档臆测。
- **业务到工程的闭环**:队员负责 AI 引擎与全栈实现,队长负责医保建模、产品与路演——从政策到可跑产品的链条在队内闭合。
- **工程可控**:36 轮 ECC 迭代,每次改动都过变更门禁(Oracle / Live / Shadow 三套隔离),规则变更留人工审批闸,不让"AI 自己改规则"失控。

### B. 监管侧价值(评委向 · 先讲透)

**全国 8600 名监管员盯 13 亿参保人 / 28.99 万家机构(美国 4 倍),飞检只查得过来 500 家;鹰眼就是替这群最缺人、最怕担责的飞检/审核员扛量的"取证放大镜"。** 对监管侧的具体用处均有产品支撑:

- **① 飞检案卷取证**:资深专家逐本翻病案找证据(单案 40 分钟)→ `/api/audit` 90 秒出三要素证据链、可点击回链原文、举证包。
- **② 优先排序·先查高风险**:面对上万案卷不知先查谁 → 稽核优先通路 `api_score` 排序,把有限飞检人力投到最高风险案卷。
- **③ 事中存疑单语义复核**:审核员逐单核"费用-医嘱-诊断"是否一致 → 只接"需核病历"的存疑单,深读非结构化案卷做语义复核。
- **④ 机构画像·抽样指导**:飞检前靠经验猜哪个科室/规则高风险 → 单案升维机构画像,指导现场抽样。
- **⑤ 误报前置过滤·省复核人力**:疑点越多误报越高、逐条人工排雷 → "引不出原文不输出" + 申诉预演,进入人工的都是"扛过一轮申诉"的,直击**第一痛点是误报不是漏判**。
- **⑥ 可审计覆盖度声明**:说不清"查了什么、没漏什么" → `coverageManifest` 把漏检率变成可签字声明,监管验收的命门。
- **⑦ 源头减负(间接)**:8600 人扛不动 28.99 万家 → 鹰眼赋能院端把历史案卷**自查干净、主动退回**("自查从宽被查从严")→ 流到飞检/审核的问题变少,**从源头替监管侧卸载工作量**(官方"关口前移/源头治本"逻辑)。

> **对监管侧金句**:*"国家'一张网'已经把'谁先被查'做到了毫秒级;鹰眼补的是下一段——飞检进场后,把那一整本非结构化案卷读成能直接写进检查结论、经得起申诉复查的证据。我们不替代稽核员、不做 AI 处罚,只做替人扛量的初筛与举证,和官方'AI 提供线索支持'的口径完全一致。"*
> **法定落地通道**:《飞行检查管理暂行办法》(国家医保局令第 6 号)允许**聘请 IT 服务机构协助飞检**——鹰眼有现成合规入口。

### C. 产品层面:七大硬核优势(均有实现支撑,可现场验证)

先把技术名词翻译成评委语言:

| 工程师说法 | 评委更该听到的说法 | 为什么更有杀伤力 |
|---|---|---|
| 多 Agent 协作 | **把部颁工作规程装进机器** | 拓扑不是我们拍脑袋设计,而是对齐复审、合议、终审和监管闭环 |
| RAG 框架 | **监管级引用,结论敢不敢签字** | 评委不关心向量库,关心每条疑点能否点到原文、页码、条款、生效日 |
| 两库规则库 | **把两库从一摞文件变成一台引擎** | 从文件、批次、条款到 checker、版本回溯、覆盖地图,能现场点验 |
| 插件化 | **安装即用、数据不出机、不动现有系统** | 政府和医院最怕大集成、打通难、数据出域;小插件绕开部署阻力 |
| 低误报机制 | **知道自己什么时候不可靠** | 证据不足转人工、干净件零误报门禁,比"AI 很准"更可信 |

**① 取证可信度三件套:合议去重 + 覆盖度声明 + 置信度传播(真护城河)**
这是鹰眼区别于"调大模型"和"国家字段规则"的工程纵深,直接解决稽核实务里的硬问题:
- **合议去重**(`reconcile`):同一笔费用被多条规则从不同角度命中时(如人血白蛋白同时触发 B-201/A-110/A-108),自动选主疑点、其余转佐证、**金额只算一次防止重复计罚**,并输出去重日志作为证明——避免"过罚"这一稽核大忌。
- **覆盖度声明**(`coverageManifest`):输出"本次覆盖了 7 个维度中的哪几个、未触发的维度是因本案无相关项(非漏查)"——把"我有没有漏"变成**可审计的声明**,正面回答监管最爱问的问题。
- **置信度传播**:OCR 低置信证据自动按比例降权、合规门禁可封顶置信度——反幻觉不是口号,是可追溯的计算。

**② 数据可信地基:把两库从"文件"变成"引擎"**
**我们不造规则、只接已公开批次**:官方全量 = **88 类规则 / 24.7 万知识点**(《2025 年版》出版物),向社会**已公开 17 批**(截至 2026-06,第十五批 6/16、第十六批 6/22、第十七批 6/29，六月连发三批)。自建爬虫(`scripts/crawl`)定向抓取国家医保局(NHSA)与省级医保局**已公开发布批次**的规则 / 知识条目、政策法规、药品目录(非平台全量接入,定位为与官方两库**同源、可对接**)。铁律:**凡未亲自验证到原文 / 下载入口的,一律标注「待核实」,绝不编造文号或链接**。

这一层的亮点不是"有 RAG",而是四个可演示的监管级细节:
- **覆盖地图**:79 条在库规则逐格展示"已实现 / 试运行 / 路线图",不把未完成伪装成已覆盖。
- **时点法**:RAG 支持按入院日 `as_of` 过滤,用案卷发生当时有效的政策版本判当时的案卷,解决飞检追溯认定最容易被申诉击穿的问题。
- **阈值来源标注**:国家给逻辑、地方增补阈值、本院自设口径分层记录,回应"两库阈值不全、地方口径不同"的现实。
- **三态文案**:同一条规则在事前对医生说人话,事中对经办说依据,事后对稽核说线索;引擎相同,身份不同,措辞不同。

**③ 精准且多元的使用场景:2 受众 × 3 阶段 = 6 格闭环**
- **框架**:**2 受众(医保监管侧 / 院端定点机构)× 3 阶段(事前提醒 / 事中结算审核 / 事后回溯监管)= 6 格价值**。**飞检 = 事后监管的一部分,结算审核 = 事中**(两拨人、两场景,勿混)。国家系统强在结构化结算数据的实时筛查,鹰眼强在深读整本非结构化案卷,在"要人逐份读"的地方替人扛量。
- **主次分明**:🟢🟢 `院端×事后自查体检`= 商业现金流首楔子 · 🟢 `监管×事后飞检取证`= 信誉锚/技术标杆 · `监管×事中`对存疑单深读复核、`院端`全三阶段做强 · `监管×事前实时拦截`= 国家 territory 诚实让位。
- **双主线**:单案卷稽核工作台(深度三要素) + 稽核优先通路(上万份案卷按 `api_score` 排序,先查高风险);`?mode=exam` 一键切院端口径。
- **覆盖 6 个临床专科方向(肿瘤 / 骨科 / 麻醉 / 重症 / 药店 / 影像)+ DRG 付费监管维度**,其中**肿瘤维度 checker 最完整、可现场端到端演示,其余专科为种子规则**(约 14 条 checker 集中在已演示维度),跳出"只会查肿瘤"的单点 demo。

**④ 多 Agent"控辩裁"= 把复审、合议、终审的工作规程装进机器**
医保没有"灰色地带"这一说法,正确机制是**申诉**(法定六环、10 个工作日死线):事中审核被退回可申诉、事后飞检判定后给申诉机会——**可申诉的是医理类争议**,医院基于**药理学/医理学**申诉,医保专家团队评定打分(行/部分行/不行),再正式回复。鹰眼的"控辩裁"本质 = **把这套争议费用申诉对质预演一遍**,前置到初筛:对高风险疑点与争议线索,启动控(稽核主张)→ 辩(机构申诉)→ 裁(三要素裁定)三方对抗,**真·LLM 路径**用位置交换裁两次防模型偏见,并为院端自动生成可用的申诉理由与举证材料。
- **诚实说明**:确定性引擎负责可复现、毫秒级的硬规则判定;控辩裁与自由文本语义判读的**真·LLM 多 Agent 路径可现场切换演示**(`?mode=llm`)。
- **价值**:把医院可预见的申诉理由前置过滤掉,**让进入人工评定的疑点都是"已经扛过一轮申诉"的**——**这直击监管侧第一痛点:误报而非漏判**(减少无效争议、省下逐条排雷的复核人力),最终评定仍由医保专家团队做。金句:**规则引擎判"像不像",控辩裁判"敢不敢拿去对质"**。

**⑤ 单租户规则自校准:沉淀人工思路、越审越准**
稽核员每一次"采纳 / 驳回 / 补材料"都被结构化沉淀:某规则在本机构被高频驳回 → 自动进**观察期(shadow,只展示不计分)**并**生成复审工单交人工裁定**(规则变更最终决定权留给人);驳回理由沉淀为 eval 草案。双阈值双链(驳回链 + 采纳链)+ overlay 不污染源规则。
- **诚实边界**:这是**单机构内的规则自校准**(数据本地、不跨院汇聚);跨机构的网络效应需在合规前提下作为远期探索,不作当下卖点。

**⑥ 安装即用:大平台拆成小插件,不动现有系统**
三入口导入:结构化 JSON、拖拽 PDF / 图片 / CSV(多模态 OCR 解析、回填 anchor 坐标)、HIS 连接器。预留医院信源接口:**MockHIS(可跑)+ FHIR R4 + HL7 v2** 适配契约。送 LLM 前 PII 脱敏,病历本地处理。部署叙事不说"打通所有系统",而说:**安装即用、数据不出机、不动你任何一个系统**。政府和医院信息化最难的是跨部门平台打通,鹰眼把能力拆成浏览器扩展、桌面哨兵、Excel 分诊、文件夹监听,先嵌进他们已经在用的流程。

**⑦ 缺失即疑点 + 知道自己什么时候不可靠**
鹰眼不把"材料不全"当作系统免责,而把它变成证据链完整度的一部分:收了手术费却找不到手术记录、用了限定药却找不到分子病理,本身就是可核查疑点或调阅线索。反过来,当关键证据缺失、OCR 置信不足、政策引用无法回链时,系统不会硬报违规,而是降级为线索 / 转人工 / 标注待核实。金句:**我们不是说 AI 永远正确,而是让系统知道自己什么时候不可靠。**

### D. 监管端插件方案:把大平台拆成飞检现场能用的小工具

监管侧不一定愿意每天打开一个新平台,但愿意在 Excel、线索台账、飞检文件夹、专家复核流程里多一个顺手按钮。监管端插件按"同一引擎、同一 KB、同一零误报红线;换输入方式 + 换输出模板 = 一个新工具"设计:

| 小插件 | 给谁用 | 输入 | 输出 | 价值句 |
|---|---|---|---|---|
| **疑点分诊器** | 基金监管处 / 经办审核员 | 智能监管导出的 Excel / CSV 疑点清单 | A 建议重点核查 / B 补材料后核查 / C 暂缓观察;每条带分档依据、可能抗辩点、建议动作 | **第一遍粗筛交给机器,人先看最值得看的。** |
| **飞检行装包生成器** | 飞检组长 / 现场检查员 | 机构名、期间、Top 疑点、案卷材料包 | 疑点清单、证据索引、现场核查任务单、缺失材料清单、询问要点 | **进场前把散材料整理成标准化任务包。** |
| **现场屏幕 OCR / 桌面哨兵** | 飞检组 IT 支撑 / 第三方协查 | 医院不给导出时的屏幕截图、复制的 Excel 行、拖入文件夹的 PDF | 秒级结构化、回填锚点、触发单案初筛 | **不给接口也能工作,数据留在本机。** |
| **申诉批阅助手** | 医保 / 医疗 / 药学专家 | 医院申诉理由 + 补充材料 | 控辩裁摘要、证据缺口、需调阅材料,不输出最终裁决 | **帮专家少翻材料,不替专家下结论。** |
| **政策 / 判例速查卡** | 法制、处罚承办、稽核员 | 规则 ID、药品、项目、关键词 | 对应两库条目、政策原文、相似公开案例、核验状态 | **写检查结论时,引用不用临时翻文件。** |
| **数据体检哨兵** | 地方规则管理员 / 信息科 | 周期性导出的结算明细或台账 | 异常科室、异常项目、疑点趋势、规则命中漂移 | **不是替代国家大数据,而是给地方口径做日常体检。** |

插件方案的边界也要写清:所有插件只做初筛、分诊、取证和材料整理;不得写"自动处罚""自动拒付""申诉成立概率";医疗合理类只提示和列证据,最终由医保专家团队和稽核员裁定。

### E. 工程可信度(敢被检验,也认边界)

不主打"测试全绿"这种结论,而主打**防自欺的机制**:Oracle ≠ Live ≠ Shadow 三套隔离(基准只用纯引擎跑,运行期治理不污染基准)、坏规则自动进观察期不计分。当前基准:AuditBench **20 案卷覆盖 7 方向、干净件零误报、违规件证据闭环**;优先通路 gold eval 7/7;RAG 在 7 条 gold query 上召回 7/7。**诚实声明**:20 案为 demo 级基准,真实案卷的泛化误报率需在试点中验证——我们建的是"怎么防止自己骗自己"的门禁,而不是"考了 100 分"。

## #15 数据资源名称、来源及覆盖范围(填表直接复制)

本作品使用的数据资源分为 **业务数据、政策知识、公开案例、评测真值、运行回归数据** 五类,全部合规可溯源,并尽量使用官方来源或官方结构。

**① 赛事提供的医保就诊结算数据 + 官方脱敏表结构**

赛事提供的医保就诊结算数据是本作品的案卷结构与结算明细底座,覆盖住院结算、费用明细、诊断、医嘱等医保稽核核心字段。我们同时参考官方脱敏表的数据结构组织方式,将病历、病程、医嘱、检验、费用清单统一编译为可追溯的案卷对象。

在此基础上,我们构造主演示案卷与 1,000 条批量筛查明细,其中预埋 32 条真值用于误报对账;实测批量分诊 1,000 条耗时 19ms,检出 32/32、误报 0。这样既保证数据形态贴近医保真实业务,又避免使用任何真实患者可识别信息。

**② 国家医保两库及政策规范公开文本**

通过自建爬虫定向抓取国家医保局(NHSA)及省级医保局已公开发布批次的规则、知识条目、政策法规、药品目录与相关规范文本。覆盖截至 2026-06 17 批公开内容,已入库 **11,027 条**:法规 56、问题清单 203、规则 10,340、目录 395、判例 29。

其中 **7,959 条带生效日期**,支持按入院日 `as_of` 回溯"当时有效"政策版本;规则的 118 处政策引用中 **106 处可解析到官方原文**,包括文号、条款、生效日和核验状态。凡未验证到官方原文或下载入口的内容,一律标注"待核实",不编造文号、不伪造链接。

**③ 国家医保局曝光台与公开典型案例**

违规样例参考国家医保局曝光台及公开通报案例构造,覆盖超限定支付、证据缺口、费用转嫁、重复收费、分解住院、DRG 高套等真实风险模式。所有样例均为虚构脱敏数据,但每个违规埋点都有底账清单,可指哪条验哪条。

**④ 案卷库、边界件与独立真值清单**

项目内置多类演示与评测案卷,覆盖肿瘤、骨科、麻醉、重症、药店、影像、DRG 付费监管、护理、康复、心血管等方向,并设置干净件、违规件、边界件三类场景。  

我们不把引擎自己的输出当"金标准",而是另设 `ground_truth_manifest` 独立声明案卷设计真值:哪些案卷应 0 疑点、哪些规则必须至少以疑点或线索命中、哪些仅作为边界观察项。该清单与引擎输出解耦,用于防止"系统自己给自己打分"。

**⑤ 运行态与回归评测数据**

包括 AuditBench、优先队列 gold eval、YHF 门禁、expected_findings 回归漂移快照、规则状态与 shadow 观察数据等。用途不是包装指标,而是监控每次规则、prompt、模型或文案变更是否引入新增误报。干净案卷零误报是 G0 红线,只要误报即不放行。

**数据合规声明**:全程不使用真实患者可识别信息;病历本地处理,送模型前 PII 脱敏;支持离线私有化部署和断网演示。数据不出机是本作品面向医保机构落地的重要前提。

## #16 开发与实现流程、关键 AI 技术、算法设计思路及核心功能模块(填表直接复制)

本作品的开发思路是:先把医保案卷变成可追溯的数据结构,再用"规则引擎 + RAG + LLM + 多 Agent harness"完成语义判读、政策引用、申诉预演和质量门禁。整体原则不是让大模型直接下结论,而是让 AI 在可审计框架内工作。

**① 开发流程:从业务闭环到工程闭环**

项目经历 36 轮"评估 → 最小修复 → 复盘"迭代。每一轮围绕真实稽核问题修正:误报、重复计罚、引用不可追溯、证据缺口、申诉理由、规则漂移等。  

工程上采用 Oracle / Live / Shadow 三套隔离:Oracle 只做基准回归,Live 承接运行态反馈,Shadow 用于观察高风险或高误报规则。"干净案卷零误报"是一票否决红线,任何规则、prompt、模型、文案变更都必须过回放门禁。

**② 总体架构:可审计语义增强层**

鹰眼不是单一模型,而是一条可审计流水线:

案卷摄取 → 事实抽取 → 规则路由 → RAG 取证 → LLM 语义判读 → 多 Agent 合议 → 三要素门禁 → 规则治理。

通俗地说:

- **确定性规则引擎**负责"像不像":药品、项目、金额、日期、机构等级等明确规则毫秒级判定;
- **RAG + LLM**负责"说不说得通":把病程、医嘱、检验等自由文本和政策条款放在同一语境里判断;
- **多 Agent harness**负责"能不能对质":控方、辩方、裁判按不同可见域分别推理;
- **合规门禁**负责"敢不敢输出":缺证据、缺引用、低置信度时自动降级或转人工。

**③ RAG + LLM 合体:不是套壳问答,而是监管级引用**

RAG 先按入院日 `as_of` 检索当时有效政策,返回文号、条款、生效日、核验状态;LLM 再只在被检索证据和案卷事实的约束范围内做语义判读。  

这解决两个传统工具做不到的问题:

1. 字段规则只能看到"用了什么药",看不懂病程里有没有真实指征;
2. 普通大模型能生成解释,但不能保证引用的是正确版本政策。

鹰眼把二者结合:既读懂自由文本,又把结论锁回官方条款和案卷原文。

**④ 多 Agent 协作:harness 工程 + prompt 工程**

控辩裁不是简单写三个提示词,而是一个带可见域隔离的 Agent harness:

- **监管指控人格**:只看规则命中、两库条款和结算数据,不能看完整病历叙事;
- **机构申辩人格**:只看临床材料和可能的医理/药理理由,模拟医院申诉;
- **专家裁定人格**:只看双方陈述书与政策弹药库,必须引用可解析知识库条目,引用不出即转人工。

Prompt 工程上,每个 Agent 有固定角色边界、证据格式、输出 JSON schema、引用要求和禁用话术;harness 工程上,系统做位置交换双跑、钢人复述、裁定一致性检查和证据缺口抽取。实测裁判位置交换一致性 100%。  

这相当于把医保流程中的"复审、合议、终审"前置预演到初筛阶段,让误报在进入人工前先被对质一遍。

**⑤ 算法与门禁设计**

- **触发器路由 + 三级短路**:79 条规则中只激活与本案相关的规则,主案卷实测激活 29 条、50 条零成本跳过;
- **三要素门禁**:每条疑点必须同时具备证据定位、政策条款、推理过程,缺一降级;
- **合议去重**:同一笔费用被多条规则命中时,只计一次金额,其他规则转为佐证,防止重复计罚;
- **覆盖度声明**:说明本次查了哪些维度、哪些未触发是因本案无相关项,把"有没有漏"变成可审计声明;
- **置信度传播**:OCR 低置信证据自动降权,关键证据不足则疑点降级为线索。

**⑥ 核心功能模块**

- 单案稽核工作台:90 秒输出"证据定位 + 政策条款 + 推理过程"三要素证据链,条条可点回原文;
- 批量优先队列:1,000 条结算明细 19ms 三档分诊,真值对账 32/32、误报 0;
- 控辩裁申诉预演:争议费用先模拟控辩裁,缺关键证据时降级为线索并列调阅清单;
- 监管/院端双前台:同一引擎,监管侧输出疑点/举证包,院端输出风险点/整改清单;
- 插件与小工具箱:Excel 分诊、飞检行装包、屏幕 OCR、医生开单浮层、编码员校验、结算员自查;
- 规则治理看板:质量门禁、覆盖地图、79/79 规则挂 checker、规则三态治理。

**一句话总结**:规则引擎判"像不像",RAG+LLM 判"说不说得通",多 Agent 判"能不能对质",门禁判"敢不敢输出"。这就是鹰眼相较传统医保审核工具的核心技术优势。

## #17 目标服务对象及拟解决的具体行业痛点(填表直接复制)

本作品面向两端用户:一端是 **医保监管侧**(医保行政部门、经办机构、飞检组、第三方协查人员),另一端是 **定点医疗机构侧**(医院医保办、医务科、病案室、编码员、结算员、临床医生、运营管理人员)。核心要解决的是医保监管"人少事多"与院端"自查查不动"的双重瓶颈。

**一、监管侧痛点:线索很多,证据很慢**

全国仅约 8,600 名医保监管员,要面对 13.27 亿参保人和 28.99 万家定点医药机构;飞检每年只能覆盖约 500 家机构,覆盖率约 0.17%。单份非结构化案卷(病历、病程、医嘱、检验、费用清单)人工初筛约 40 分钟。  

国家系统已经很强地解决"谁该被查",但飞检和复核现场仍要回答:"证据在哪?适用哪条?有没有漏?医院申诉时能不能扛住?"这部分案卷级语义取证仍高度依赖人工,是全链条最耗人力的空白。

鹰眼面向监管侧提供:

- 对稽核员:90 秒三要素证据链,减少翻病历和写结论时间;
- 对飞检组长:飞检行装包、Top 疑点、证据索引、缺失材料清单;
- 对经办审核员:Excel / CSV 疑点分诊,A 重点核查 / B 补材料 / C 暂缓观察;
- 对医保专家:申诉批阅助手,生成控辩裁摘要、证据缺口、需调阅材料;
- 对规则管理员:阈值试算、规则三态治理、shadow 观察和回放门禁;
- 对第三方协查人员:屏幕 OCR、桌面哨兵、断网本地运行,不要求医院开放接口。

**二、院端痛点:自查是强制动作,但语义层查不动**

2026 年医保自查自纠扩至 12 个重点领域、回溯两年,官方口径是"自查从宽、被查从严"。但真实差距是,某三甲医院飞检查出 82 项违规、涉及 3,466.7 万元,此前自查仅退回 3.23 万元。这个千倍缺口说明,医院不是不想查,而是现有工具多停在结构化编码质控层,读不懂整本病历和费用之间的语义关系。

鹰眼面向院端提供:

- 对医保办/医务科:12 领域历史案卷批量体检、整改清单、主动退回金额测算;
- 对病案室/质控人员:病历材料缺口、诊疗记录与费用项目不一致提示;
- 对编码员:DRG/DIP 高套、错编、分组异常前置校验;
- 对结算员:出院前整单自查,发现风险点后补材料或转人工;
- 对临床医生:开单浮层提醒,政策限定类红色拦截,医疗合理类黄色提示并允许说明理由;
- 对运营管理者:遵从率、风险金额、整改进度、复检对比看板。

**三、两端 × 三阶段:鹰眼具体做什么**

鹰眼不是单点工具,而是同一台引擎在"监管侧 / 院端"两端、"事前 / 事中 / 事后"三阶段中的复用。监管侧叫疑点、线索、举证包;院端叫风险点、整改清单、复检对比。

**事前提醒阶段**:监管侧不直接替代医生开单,重点关注高风险趋势、规则口径和政策解释;院端则在医生开单、科室用药、耗材使用前获得提醒。鹰眼提供医生开单浮层、政策速查卡、红黄灯提示:政策限定类能硬拦的硬拦,医疗合理类只提示并允许医生填写理由,既减少未来疑点,也不粗暴干扰正常诊疗。

**事中审核阶段**:经办审核员面对智能监管导出的疑点清单,需要快速分诊和复核;院端编码员、结算员也需要在提交前做 DRG/DIP、费用与医嘱一致性自查。鹰眼提供 Excel/CSV 疑点分诊器、结算前整单自查、编码员校验、A/B/C 三档分诊,把低价值误报提前降噪,让人先看最值得看的单。

**事后监管阶段**:飞检组和稽核员要逐份读案卷、找证据、写报告、处理申诉;院端医保办和医务科要回溯两年历史案卷,形成整改台账。鹰眼提供单案三要素证据链、飞检行装包、控辩裁申诉预演、整改前后复跑、主动退回金额测算,把"翻材料"变成"审证据",把"填表自查"变成可核验的整改产物。

**四、行业意义:源头减负,而不是制造更多告警**

院端把旧账查干净、医生开单时被及时提醒,未来流到监管侧的疑点就更少、更实;监管侧通过鹰眼把有限人力投向真争议、真违规,避免被低质量告警淹没。  

鹰眼定位为初筛与取证辅助工具,不做 AI 自动处罚,所有定性、追回与处罚仍由稽核员、医保专家和医保部门最终裁定。该定位与官方"AI 提供线索支持"口径一致,也符合《飞行检查管理暂行办法》中 IT 服务机构协助飞检的落地通道。

# 二、作品在线文档(#14 · 飞书 · 可直接发布的完整长文)

> 飞书目录:00 一句话 → 01 业务价值 → 02 AI 创新性 → 03 技术实现 → 04 交互与设计 → 05 能力边界与责任 → 06 落地与团队。下文每节均为成稿。

## 00 · 一句话

**鹰眼 EAGLEEYE 是医保基金稽核的 AI 初筛员**:它读取、解析病历、医嘱、费用清单等非结构化案卷,在 90 秒内做"费用—诊断—医嘱"三方语义交叉,输出每条都带「证据定位 + 政策条款 + 推理过程」三要素、可直接拿去对质的稽核报告。它用多 Agent"控辩裁"预演飞检/审核现场的**争议费用申诉**,坚持"引不出原文的疑点不输出",最终评定由医保专家团队做。它站在国家医保两库肩上,补结构化大数据筛查之外那一段——**单份非结构化案卷的深度语义取证**。

## 01 · 业务价值

### 1.1 背景:救命钱与监管现实

医保基金是老百姓的"救命钱"。据国家医保局统计,**2025 年全国追回违规资金约 342 亿元**,查实欺诈骗保机构 1626 家——监管体系在高效运转,也说明存量违规规模仍然巨大。

**但真正的瓶颈是"人少事多,谁能解决这个问题,谁就是胜者"。** 全国医保监管员仅 **8600 人**,要盯 **13 亿参保人 / 28.99 万家机构**,人均监管量是**美国 4 倍**,缺口大到招人补不上;2025 追回 342 亿靠拼人海 + 抽查(2024 飞检只查得过来约 500 家),单案人工初筛约 **40 分钟**。**鹰眼 = 打破"人少事多"瓶颈的 AI 人力倍增器**——不替代人,替人扛掉海量非结构化案卷初筛(40 分钟→90 秒),让稀缺专业人力只花在真争议、真违规上。

随着《医疗保障基金使用监督管理条例》(国务院令第 735 号,行政法规)及其实施细则(国家医保局部门规章,2026-04-01 施行)落地、飞行检查"四不两直"常态化、定点机构自查自纠在 12 个重点领域全国铺开,**对"非结构化案卷深度判读"的需求正在快速放大**。《医保基金监督检查五年行动计划(2026—2030)》亦将"探索人工智能 + 病例判读"列为方向。

### 1.2 国家系统强在哪、留白在哪

国家两库(规则库 + 知识库,约 88 类 / 24.7 万条)与大数据智能监管,擅长**结构化结算数据的大规模筛查、可编码规则、事前提醒与事中审核**,解决"谁该被查"。但以下仍是结构上的留白:
- **非结构化案卷**:飞检现场是扫描件、PDF、多系统拼包的病历 + 病程 + 医嘱 + 检验,需逐份深读;
- **语义级交叉验证**:把"费用—诊断—医嘱"对上(如靶向药与基因检测证据是否匹配),字段规则做不到;
- **案卷级取证**:从"疑点 flag"到"可写进报告、可拿去对质"的三要素证据链。

**鹰眼正是补这两段。** 国家系统加速"谁该被查";鹰眼做"单份案卷怎么在 90 秒内初筛、怎么写出证据链"。两者是上下游互补,不是重复建设。

### 1.3 三类核心场景

| 场景 | 用户 | 痛点 | 鹰眼价值 |
|---|---|---|---|
| 飞检 / 专项稽核 | 医保行政部门稽核员 | 单份案卷人工约 40 分钟,时间耗在翻材料、对条款 | 90 秒初筛 + 三要素证据链 + 举证包,人力只花在真争议 |
| 院端日常自查 | 定点医疗机构合规 / 医务部门 | 12 领域 × 两年历史案卷翻不完;不知改哪、怕"说了改了没证据" | 按风险排序历史案卷;体检模式给带条款引用的整改清单;嵌入日常病案质控 |
| 争议申辩 | 医院 / 稽核双方 | 争议项要能回链原文,临时翻病历成本高 | 三要素 + 控辩裁留痕,申辩材料现成 |

> 关键洞察:自查报送之后不是闲置期,而是**监管验收期**——飞检会核查"自查质量"。鹰眼让自查从"填表"变成"可核验的产出"(台账 + 金额 + 条款 + 整改记录)。一句话:**国家两库帮医院少犯新错;鹰眼帮医院把旧账查清、查清能写进报告、改完能证明。**

### 1.4 商业化路径(分阶段、诚实)

> **商业落地首楔子 = 院端自查体检层。** 院端自查**不是我们创造的需求,是政策强制 + 官方鼓励的**:国家医保局自查自纠 2025 覆盖 9 领域 → **2026 扩 12 领域**、回溯 2 年,不做就挨罚;官方明文"**自查从宽、被查从严**"、鼓励定点机构主动对接两库自查自纠,《五年行动计划(2026—2030)》更把"**源头治本、关口前移**"列为方向——**卖院端自查 = 帮官方做它要做的事**。院端**不是不想查、是没能力查**(现普遍靠 Excel 人工筛),付费驱动是恐惧不是合规理想:一次飞检罚款百万至千万,工具费是零头,**ROI 锚定"避免一次飞检损失"**。且院端自查做好 = 直接减少监管侧工作量(§B⑦),两侧同一根引擎。

- **近期**:监管侧轻量 SaaS 稽核辅助 + 院端"自查体检"服务;数据本地、可离线、送 LLM 前 PII 脱敏;
- **冷启动**:不依赖大规模真实病历训练,而是**规则驱动 + 专家标注少量金标案例**(低数据依赖,合规友好);
- **种子客户**:优先借助团队医疗行研 / 咨询人脉与顾问的行业资源,争取三甲医院医务科 / 医保办的试用;
- **远期**:随规则资产与方法论沉淀,探索按追回 / 节约金额的效果付费。

## 02 · AI 创新性(AI 在哪里、为什么是核心)

鹰眼不是"把规则写成 if-else",也不是"把病历丢给大模型问一句"。AI 与工程在四个层次协同,每一层都为"可信"服务:

**① 三要素证据链 + 反幻觉门禁(地基)**
每条疑点必须给出①证据定位(费用行号 / 病历单据 + 原文摘录)②政策条款(KB 引用 ID + 核验状态)③推理过程。三要素门禁缺一降级——**引不出原文的疑点不输出**。这把大模型幻觉关进证据的笼子。

**② 多 Agent"控辩裁"= 争议费用申诉机制预演(核心创新,诚实切分)**
医保没有"灰色地带",正确机制是**申诉**(基于药理/医理,医保专家团队评定打分)。对高风险疑点与争议线索,启动控 → 辩 → 裁三方对抗,真·LLM 路径用位置交换裁两次防偏见。这把只有飞检/审核现场才发生的**申诉对质**前置到初筛:该硬的硬、该降级的降级、该补材料的列清单,并为院端生成可用的申诉理由与举证材料。
> **诚实分工**:确定性引擎负责可复现、毫秒级的硬规则(demo 默认路径);自由文本语义判读与控辩裁的真·LLM 多 Agent 路径按需启用(`?mode=llm`)。我们主动区分"确定性可算的"与"只有读病历自由文本才能判的",前者保证可复现与低成本,后者才上模型。
> **与传统智能审核的差异**:传统输出"违规 flag";鹰眼输出"flag + 正反论证 + 裁定理由",误报事前模拟申诉、能降级先降级,争议区显式区分 疑点 / 线索 / 不输出。

**③ 取证可信度三件套(国家字段规则给不了的工程纵深)**
- **合议去重**:一笔钱被多规则命中 → 合并主疑点 + 佐证、金额只算一次(防过罚),输出去重日志;
- **覆盖度声明**:输出"覆盖了哪些维度、未触发是因本案无相关项而非漏查"——把"有没有漏"变成可审计声明;
- **置信度传播**:OCR 低置信证据降权、门禁封顶置信度。

**④ 语义级三方交叉 + 单租户自校准**
语义的真实战场在**自由文本**:不是"分子病理/基因检测证据字段为空"(那是结构化判空),而是读懂病程里"家属要求靶向""建议外购"这类自然语言不构成用药指征、并带否定词识别(排除"无呼吸衰竭"误判)。复核反馈结构化沉淀 → 高误报规则自动进观察期 → 生成复审工单交人裁定 → 驳回理由沉淀为 eval 草案。规则在**单机构内**自校准、越审越准。

**诚实的工程化 AI**:默认走确定性引擎(可解释、可复现、毫秒级),LLM 为可选语义增强;无 Key 时如实标注回退、绝不假跑。我们不信任"模型一把梭",所以造了这条可审计的流水线。

## 03 · 技术实现(整体思路)

### 3.1 六层流水线 + 横向门禁

```
①摄取层  多格式导入(JSON/PDF/图片/CSV)+ 多模态 OCR + HIS 连接器 → 回填 anchor 坐标
②事实层  编译为 Case Object,每条事实自带源锚点 + OCR 置信度
③路由层  触发器路由 + 三级短路,只激活相关规则,其余零成本跳过(省 LLM 调用)
④判定层  确定性规则 checker(6 专科+DRG)+ 合议层(多规则命中→合并主疑点+佐证、金额去重)
⑤门禁层  合规前置门禁(三要素降级、措辞约束)+ 控辩裁 + CoVe 取证自检 + 置信度校准
⑥治理层  三态规则状态机(在役/观察/下线)+ 误报回流 + eval 草案沉淀(变更留人工闸)
```

### 3.2 "语义增强层"到底是什么(给个硬定义)

> **语义增强层 = 把国家两库里"可编码的字段规则",通过 RAG 检索(按 `as_of` 取当时有效政策)+ 跨文档勾稽规则 + 可选 LLM 自由文本判读,扩展到非结构化病历上做"费用—诊断—医嘱—检验"的一致性判定。**

它不是套壳大模型,也不是纯关键词匹配:结构化能确定的用确定性引擎算(可复现);只有读自由文本才能判的才上 LLM。

### 3.3 知识库工程

爬虫(`scripts/crawl`)定向抓取国家 / 省医保局已公开发布批次的两库条目、政策法规、药品目录;逐条核验 URL 与原文,三处交叉验证,每条带核验状态,**未核实即标注、不编造**。RAG 在 gold query 上召回 7/7(G4),支持 `as_of` 入院日政策版本过滤。

### 3.4 优先通路评分

```
tier      = 1(含 active 疑点) / 2(仅线索) / 3(无 active)
core      = 几何均值(EC 证据强度, AMT 金额, SEV 严重度)
api_score = round(100 × core × HistoryPrior × Breadth × Outlier, 1)
排序:tier ASC → api_score DESC;shadow 永不计分,沉底单独展示
```

### 3.5 工程纪律(为什么敢被检验)

`rules.yaml` 单一事实来源 → `npm run build:rules`;**Oracle ≠ Live ≠ Shadow 三套隔离**,基准只用纯引擎跑,运行期治理不污染基准;G0 红线 = 干净件误报 0;任何规则 / prompt / 模型变更都过门禁;CI 自动跑。技术栈:零依赖 Node.js HTTP 服务 + 确定性引擎 + 静态前端(刻意轻量、可审计),可选 LLM(MiniMax / Anthropic)、Supabase 向量检索、PP-Structure OCR sidecar。

## 04 · 交互与设计

- **稽核工作台**(`/`):左栏材料包、右栏疑点卡。单案七步:导入 → 选案 → 稽核 → 审阅疑点卡 → 点证据跳原文高亮 → 采纳 / 驳回 / 控辩裁 → 导出核查清单。
- **稽核优先通路**(`/priority.html`):上万案卷按 api_score 排序,tier / 分桶展示,shadow 沉底;批量稽核、举证包、违规统计。
- **项目看板**(`/dashboard.html`):YHF 门禁、AuditBench、规则治理三态、工程成熟度——一屏看清"产品有多稳"。
- **双模式**:稽核模式(监管口径:疑点 / 建议追回) vs 体检模式(院端口径:风险点 / 主动退回),引擎相同、仅换措辞,一键切换。
- **视觉**:瑞士国际主义风格,克莱因蓝主色,信息密度高、可打印导出。

## 05 · 能力边界与责任(主动声明)

### 5.1 能力边界

| 我们能做 | 我们暂不做 / 需说明 |
|---|---|
| 单份案卷三方交叉 + 三要素输出 | 不替代国家智能监管的全量结算拦截 |
| 有独立 checker + 案例的规则真 fire | 规则库 79 条,47 个确定性 checker,其余走族级/批量/映射接入 |
| 扫描件 / 表格语义初筛(确定性 + 可选 LLM) | 无 Key 时不跑真·LLM 语义(回退确定性) |
| 优先队列 + 批量稽核 + 举证包 | 生产级多用户鉴权、机构画像多时间断面仍在完善 |
| 20 案基准、干净件 0 误报、三套隔离门禁 | 真实案卷泛化误报率需试点验证;非"考了 100 分" |
| 控辩裁·争议费用申诉预演 + 单租户规则自校准 | 跨机构数据汇聚需合规前提,为远期探索 |

### 5.2 责任边界(监管采购最关心)

鹰眼是**初筛与取证辅助工具,所有定性与处置由稽核员 / 飞检组最终裁定**,产品不做 AI 自动处罚。作为初筛工具,我们以**高精确率(低误报)为先**——把"看起来像但证据不足"的降级为线索而非硬报,避免误伤合规诊疗、避免无效告警淹没稽核员;漏判的兜底交给批量优先通路的全量扫描、人工复核与国家全量结算大数据。**禁止夸大**:不替代国家系统、不做自动处罚、不宣称零漏报。我们主动讲清边界——对监管而言,**一个知道自己边界的工具,才敢被信任**。
