## 一、快速上手（评委 / 首次演示）

### 1.1 产品一句话

鹰眼为稽核员提供 **90 秒级** 案卷初筛：读取费用清单、病历、医嘱等非结构化材料，做「费用—诊断—医嘱」三方交叉验证，输出带 **三要素证据链** 的结构化报告。**引不出原文的疑点不输出**（高精确率（低误报）为先 + 漏判兜底：优先通路全量扫描+人工复核+国家大数据）。

**定位**：事后飞检 / 专项稽核 / 案卷复核 — **不是** 院端实时拦截或经办结算引擎。

### 1.2 一分钟启动

```bash
cd prototype/app
node server.js
# 浏览器打开 http://localhost:3700
```

| 入口 | URL | 用途 |
|---|---|---|
| 稽核工作台 | http://localhost:3700/ | 单案卷稽核、三要素、控辩裁 |
| 稽核优先通路 | http://localhost:3700/priority.html | 飞检批量优先队列 |
| 项目看板 | http://localhost:3700/dashboard.html | YHF 门禁、AuditBench、工程成熟度 |

在线预览：https://yingyan.vercel.app/

### 1.3 90 秒 Demo 脚本（肿瘤 NSCLC 主线）

1. **左栏**展示患者材料包（病案首页、病程、医嘱、检验、费用清单等）— 默认案卷 `main`（NSCLC 化疗住院）。
2. 点击 **「开始稽核」**：引擎跑规则路由，毫秒级出报告（人工约 40 分钟 vs 鹰眼约 1–5 ms 确定性路径）。
3. 展开 **5 疑点 + 2 线索**（疑点金额约 ¥6,301，另 2 线索 ¥3,780），逐条看三要素：
   - ① 证据定位（可点击跳转费用行 / 病历锚点）
   - ② 政策条款原文（KB 引用 ID + ✅已核验 / ⚠待核验）
   - ③ 完整推理过程
4. **重点讲 T-201 奥希替尼**：靶向药限 EGFR 突变 — 全材料无分子病理/基因检测证据 → 证据闭环疑点。
5. **重点讲控辩裁降级**：T-205 升白针 — 辩方指出前次血常规不在本包内 → 疑点降为线索（高精确率（低误报）为先 + 漏判兜底的工程实现）。
6. **主动讲两个「正确不报」**：贝伐珠单抗（非靶点类，不报 T-201）、放化疗周期再入院（白名单，不报 C-301）。

### 1.4 可选：开启 LLM 语义路径

## 二、稽核员操作手册

### 2.1 核心概念

| 概念 | 说明 |
|---|---|
| **案卷 (AuditCase)** | 一次就诊的完整材料包 |
| **疑点** | 三要素证据闭环，可直接对质 |
| **线索** | 证据不全，需补材料或人工复核 |
| **不输出** | 无足够证据或命中除外情形 |
| **shadow finding** | 影子规则命中，**不计入**优先分数与 suspected_count |

### 2.2 单案卷稽核流程（工作台）

```
导入材料 → 选择案卷 → 开始稽核 → 审阅疑点卡 → 采纳/驳回/控辩裁 → 导出核查清单
```

#### 步骤 1：导入材料

三种入口（顶栏）：

- **结构化 JSON**：直接加载 demo 案卷
- **材料导入 / 快速导入**：拖入 PDF、图片、CSV、JSON → `POST /api/intake/batch`
- **HIS 连接器（Mock）**：`GET /api/connectors`

可选 L1 解析 sidecar（PDF bbox 坐标）：

```bash
cd prototype/ppstructure && bash run.sh   # 终端 1
cd prototype/app && node server.js        # 终端 2
```

健康检查 `GET /api/health` 中 `ppstructure.reachable` 为 true 表示 L1 就绪。

#### 步骤 2：开始稽核

- 按钮：**开始稽核**
- API：`POST /api/audit`（body 可选 `{ caseId, record, inject }`）
- 模式：`?mode=exam` 切换 **体检模式**（措辞变为「风险点 / 院端整改」，引擎相同）

#### 步骤 3：解读疑点卡

每条 active finding 含：

| 要素 | 操作 |
|---|---|
| 证据定位 | 点击 ⚓ 跳转原件高亮（消费 Case Object anchor） |
| 政策条款 | 展开 KB 引用，查看 `verify_status` |
| 推理过程 | 展开推理链；CoVe 可生成取证自检问题 |
| 控辩裁 | 点击 ⚔ 触发控/辩/裁三方对质（`POST /api/debate`） |

#### 步骤 4：导出文书

- **疑点核查清单**：`GET /api/export/checklist`
- **机构画像汇总**：`GET /api/export/institution`（mock 数据）

### 2.3 稽核优先通路（飞检批量）

入口：**http://localhost:3700/priority.html**

适用场景：一次导入上百份案卷，需按 **稽核优先指数 (api_score)** 排序，先查高风险。

#### 优先队列逻辑

```
tier = 1 若含 active 疑点；2 若仅线索；3 无 active
core = 几何均值(EC, AMT, SEV)   # 默认权重见 priority-score.js
api_score = round(100 × core × HistoryPrior × Breadth × Outlier, 1)
排序：tier ASC → api_score DESC → S DESC
```

**shadow findings 永不进 api_score**，沉底 **shadow 桶**单独展示；**boundary 桶**供 AuditBench 边界件观测（iter-36）。

#### 优先通路 v2 能力

| 能力 | API / 模块 |
|---|---|
| 违规性质 + 处置建议 | `priority-nature.js` |
| DRG L3 线索 / 首页一致性 | `priority-enrich.js` |
| 特例单议离群抑制 | `priority-score.js` |
| 举证包 HTML/PDF | `POST /api/evidence-package` |
| 违规点统计表 | `GET /api/report/violation-summary` |
| 9 大领域 risk_tags 加权 | `priority-enrich.js` + `/api/priority/config` |
| 自查清单 | `GET /api/checklist` |
| 历史核查 / 命中率 | `GET /api/history` |

#### 批量操作

1. 导入：`POST /api/intake/batch` 或 `/api/ingest`
2. 查看队列：`GET /api/priority/rank`
3. 勾选 Top-N → `POST /api/audit/batch` → 写入 `audit_records` → **HistoryPrior** 更新

数据落盘：`prototype/data/priority/store.json`

#### 体检模式

优先通路页勾选 **体检模式**：仅改展示口径（「风险点」替代「疑点」），**api_score 不变**。

### 2.4 双模式对照

| 维度 | 稽核模式 | 体检模式 |
|---|---|---|
| 受众 | 医保行政部门稽核员 | 医院自查 / 飞检前置 |
| 输出措辞 | 疑点 / 责令退回 | 风险点 / 主动退回 |
| 引擎 | 相同 | 相同 |
| 参数 | 默认 | `?mode=exam` 或 UI 切换 |

### 2.5 看板（工程 / 管理视角）

`dashboard.html` 提供：

- YHF 门禁指标（`GET /api/yhf`）
- AuditBench 20 案卷指标（`GET /api/bench`）
- 规则治理 / shadow 状态（`GET /api/rule-governance`）
- 优先通路摘要（`#priority` 锚点）

### 3.5 验收与回归

```bash
# 变更门禁（CI 同款）
bash yhf/run.sh --strict

# 优先通路
node scripts/verify-priority-pathway.js
node scripts/run-priority-gold-eval.js

# 看板前端契约
node scripts/verify-dashboard-frontend.js

# Prompt 评测（需 eval/.env）
cd eval && bash run_v7.sh
```

### 3.6 YHF 工程纪律

1. **Oracle ≠ Live ≠ Shadow** — bench/gate 只用 Oracle
2. **G0 红线** — 干净件误报 = 0
3. **rules.yaml** 为规则单一事实来源 → `npm run build:rules` 生成 rules.json
4. 任何 prompt / 规则 / 模型变更应过 gate

### 3.7 ECC 式迭代（Evaluate → Correct → Continue）

每轮迭代（见 `prototype/docs/ITERATION_REPORTS.md`）：

1. **Evaluate**：跑 YHF / verify / bench，记录缺口
2. **Correct**：最小 diff 修复
3. **Continue**：写 KPT 复盘 + 下轮 SMART 目标

优先通路 v2 专用 eval：`.claude/evals/priority-pathway-v2.md`

## 四、能力边界对照表

| 我能做 | 我暂不能 / 需说明 |
|---|---|
| 单份案卷三方交叉验证 + 三要素输出 | 替代国家智能监管全量结算拦截 |
| 有独立 checker + 案例 的规则真 fire | 规则库 79 条,47 个确定性 checker,其余专科条目走族级/批量/映射接入 |
| 扫描件/表格语义初筛（结构化 demo + 可选 LLM） | 无 Key 时不跑真·LLM 语义（回退确定性） |
| 优先队列 api_score 排序 + 批量稽核 | 生产级 Postgres / 多用户鉴权（Phase 8） |
| AuditBench 20 案、干净件 0 误报红线 | 全国 88 类规则库全量抓取（BACKLOG B03） |
| 对抗注入 E-503 检测 | 完整 OCR bbox（需 L1 sidecar + Key） |
| 举证包 / 违规统计 / 自查清单 | 机构画像多时间断面趋势（Phase 7 预留） |
| 控辩裁 + eval_draft 写回（iter-36） | 采纳/驳回全链路持久化闭环（部分 UI 仍 mock） |

**现场建议**：Demo 使用 `main`（肿瘤）、`clean`（零误报）、`ortho`/`drg` 案卷；勿切换到无 checker 的专科案卷 expecting 违规输出。
