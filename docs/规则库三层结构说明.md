# 鹰眼规则库：62 + L3 + ZB + 埋点 = 79

本文解释**为什么平台显示 79 条规则**，以及与你熟悉的 **13 族 62 条分段**如何对应。不改对外口径，只把账算清楚。

---

## 一、四个数字各是什么

| 数字 | 名称 | 含义 |
|------|------|------|
| **62** | 13 族命名体系 | F/A/B/C/D/E/T/M/ICU/P/IMG/CV/BP 前缀下的 rule_id（你在分段表里加总的那 62 条） |
| **+4** | 埋点/跨族扩展 | SUR-401（手术折价）、TRACE-101（追溯码）、NUR-303（护理天数）、AGE-101（年龄用药） |
| **+5** | L3 族引擎 | L3-DRX / L3-CDM / L3-SAF / L3-TCM / L3-DS（一条引擎覆盖多条国家 GZ） |
| **+8** | ZB 批量监测 | ZB-001～ZB-008（药费占比、再入院率等，需批量结算数据） |
| **=79** | rules.yaml 总条数 | 也是国家两库 **71 GZ + 8 ZB** 框架格数 |

```
62（13族）
 + 4（埋点）
 + 5（L3族）
 + 8（ZB批量）
─────────────
 79（rule_id 总数 = 国家框架格）
```

## 三、不在 13 族里的 +17 条

ZB-001 药费占比 … ZB-008 非计划再入院率。执行入口：`batch_settlement_rows` + `stats-monitoring.js`。

## 四、79 条 vs 国家 GZ/ZB 框架

- **71 条 GZ**：政策/管理/医疗类，在 `coverage-map` 上占 71 格。
- **8 条 ZB**：统计指标，占 8 格。
- 多条 GZ **复用**同一鹰眼 rule_id（例如 7 条 GZ → F-002），所以 **79 格 ≠ 79 份独立代码**。

---

## 五、引擎执行层（深度实装）

每条 rule_id 在 `audit-engine.js` 的 `ruleCheckers` 中应有对应函数（含 B-201-IND 走适应症索引、ZB 走批量引擎）。

- **62 族内**：原先 25 条仅 YAML 声明 → 已由 `extended-rule-checkers.js` 补齐 checker。
- **+17**：L3/ZB/埋点/SUR 等原有 checker 或专用路径。
- **干净案卷**：无 planted 字段、无违规结构化信号 → **零误报**（G0 门禁）。
