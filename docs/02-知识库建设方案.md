# 鹰眼 · 两大知识库建设方案与资料来源清单 v0.1

> 对应产品技术路线中的"医保目录与支付规则 RAG 知识库"。本方案拆为 KB1（医保审核规则库）与 KB2（临床指南库），并给出黑客松周期内的抓取优先级。
> 编制日期：2026-06-12。所列文件均为公开发布的官方/权威文件，私有化部署内部使用，引用时标注出处。

---

## 0. 总体架构

- **KB1 回答"违不违规"**（政策、目录、规则、负面清单）；**KB2 回答"该不该做"**（入院指征、用药指征、诊疗规范）。两库职责分离，Agent 推理时分别取证。
- 检索策略：KB1 以**结构化精确召回**为主（目录按药品行、法规按条款、清单按问题项），保证"引得出原文"；KB2 以**病种/药品为键的向量检索**为主。
- **引用 ID 规范（强制证据链的工程基础）**：`[KB1-文档代号-条款定位]`，如 `[KB1-条例-第38条(三)]`、`[KB1-目录2025-西药-XA01-备注]`。Agent 报告里的每条政策依据必须落到一个引用 ID + 原文，否则三要素门禁不放行。

### 1.2 入库 Schema（KB1）

```json
{
  "doc_id": "KB1-条例",
  "layer": "法规|目录|规则|清单|案例",
  "authority": "国务院|国家医保局|省医保局",
  "doc_no": "国务院令第735号",
  "effective_from": "2021-05-01",
  "effective_to": null,
  "region": "全国|某省",
  "unit_type": "条款|药品行|项目行|问题项|案例",
  "locator": "第38条(三)",
  "text": "……原文……",
  "violation_tags": ["重复收费"],
  "linked_rules": ["A-101", "A-102"]
}
```

分块原则：法规按**条**、目录按**药品/项目行**、问题清单按**问题项**、案例按**案件**。一律保留原文与定位，禁止入库改写版（改写只放在规则库的 trigger_logic 里）。

### 1.3 黑客松抓取顺序（P0 → P2）

1. P0（第1周）：条例 + 实施细则全文；2025版药品目录"备注"列；2026年问题清单（demo 科室对应领域）；两库1.0附件。
2. P1（第2周）：锚定省份价格项目目录与负面清单；立项指南相关专业分册；曝光台案例 20 件。
3. P2（演示前补强）：耗材代码、DRG/DIP 2.0 分组方案（若纳入 D 类规则）。

### 2.2 黑客松收敛策略：按"病种包"组织

不要全量建库。确定 demo 场景后，每个场景建一个病种包（建议 2–3 个）：

```
病种包 = 临床路径要点（入院指征/必查项目/核心治疗）
       + 该病种高频药品说明书摘要（适应症、用法用量）
       + 对应医保目录限定支付条款（与 KB1 行级互链）
       + 该专科问题清单条目（与 KB1 互链）
```

### 2.3 起步样例：高频"限定支付"药品核对卡（演示素材）

> 下表为常见限定方向的**示意**，列出 Agent 的核对要素；**入库前必须以 2025 年版目录"备注"列原文逐条校准**（限定条款历年有调整）。

| 药品（类） | 常见限定方向 | Agent 核对要素 |
|---|---|---|
| 人血白蛋白 | 限抢救、重症或低白蛋白血症等 | 病程是否记载抢救/重症；白蛋白检验值 |
| 质子泵抑制剂注射剂 | 限有相应诊断且不能口服的患者 | 是否禁食/吞咽困难记录；可否口服 |
| 司美格鲁肽等 GLP-1 | 限 2 型糖尿病相应适应症（减重使用不支付） | 糖尿病诊断与血糖证据 |
| 抗肿瘤靶向药 | 多数限基因/靶点检测阳性或特定线数 | 基因检测报告；既往治疗线数 |
| 中药注射剂（部分） | 限二级及以上机构、限重症/急性期 | 机构等级；病情严重程度记录 |
| 静注人免疫球蛋白 | 限特定免疫缺陷/危重情形 | 相应诊断与检验 |
| 生长激素 | 限儿童特定疾病 | 年龄、诊断、检测值 |
| 重组人促红素 | 限肾性贫血等 | 肾功能、血红蛋白 |

## 3. 抓取与合规注意

1. 官方附件优先（PDF/Excel 可直接结构化）；官微公开的清单常为**长图**，需 OCR → 人工校对一遍再入库（清单措辞将直接进稽核报告，错字不可接受）。
2. 政府法规与公文可全文入库；**学会指南/教材有版权**，做条目级摘引并保留出处字段，演示与部署均为内部使用。
3. 版本治理：目录年度调整（2025版自2026-01-01执行）、"两库"年度动态调整、问题清单逐年扩领域 → 每条知识必须带 `effective_from/to`，Agent 取证时按**就诊日期**选版本（稽核的是历史行为，不能拿新规则套旧行为）。
4. 数据不出域：知识库全部为公开政策文件，可随私有化包分发；患者病历数据只在本地处理，二者物理分离。

## 4. 与 Agent 的接口约定

- 知识库检索返回：`{ref_id, 原文, 定位, 效力级别, 生效区间}`，缺任一字段视为检索失败。
- 报告中政策依据仅允许引用 KB 返回的原文，禁止 LLM 凭记忆生成条款文本（防幻觉条款）。
- KB1 与 KB2 交叉互链（限定支付条款 ↔ 说明书 ↔ 问题清单条目），供 Agent 一跳取齐证据。
