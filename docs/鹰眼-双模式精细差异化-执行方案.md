# 鹰眼 · 监管稽核 vs 院端自查 双模式精细差异化 · 执行方案

> 交接对象：执行 AI / 前端工程。读完本文即可动手，不需要再问需求方。
> 日期：2026-07-04 · 基于当前 main 工作区代码（app.js 约 2723 行版本）。

---

## 0. 背景 · 问题 · 总原则

### 0.1 问题诊断

主平台 `index.html` 一页承载两个模式：监管稽核（`MODE='audit'`，蓝皮）与院端自查（`MODE='exam'`，绿皮）。
当前差异只有：换色、部分文案替换、规则子集过滤、多一个「登记整改」Tab。
**信息对象、层级、动作、度量完全相同**，所以观感上是"同一个软件换了皮肤"。

### 0.2 设计总原则（执行时时刻对照）

1. **不动双栏几何**。`layout-two` 左右双栏布局一个像素不改，禁止上下分块。差异全部做在"同一副骨架、不同器官"上。
2. **角色叙事驱动一切差异**：
   - 监管 = **取证台**。动词：查实、举证、对质、调阅、移送。终点产物：举证包。核心度量：证据三要素齐备率、涉嫌金额。**监管永远不打分**——监管是取证不是评分。
   - 院端 = **体检报告 + 整改驾驶舱**。动词：暴露、自查、整改、留痕、主动退回。终点产物：整改清单 + 复跑 diff。核心度量：体检总分、整改进度、暴露金额降幅。
3. **只改前端**。所有改动限于 `prototype/app/public/` 下的 `index.html` / `app.js` / `style.css`。不碰 `engine/`、`server.js`、`rules.yaml/json`、任何 `data/`。
4. **诚实原则**：新增的分数/比率必须由已有真实数据合成，构成公式对用户透明（悬停可见），不得编造。G0 干净件零误报红线不受影响。
5. **表述红线（词汇表）见 §7**，所有新增/改动文案必须逐字取自 §7，不许自由发挥。

## P1 · 院端体检总分（最高优先级，一眼分辨两个产品）

### 1.1 需求

院端模式跑完自查后，报告 hero 区替换为「体检报告头」：一个 0-100 的**合规体检总分**圆环 + 体检等级 + 三个分项小分。监管模式 hero 维持现状（P4 再小改）。

### 1.2 打分公式（全部来自已有数据，前端计算）

新增函数 `examHealthScore(report)`，放在 `reportHeroHTML` 附近：

```
输入：REPORT.report_meta.summary (s)、EV_CHAIN、rectProgress(REPORT)
分项1 费用合规分 fee   = max(0, 100 - 12×s.suspected_count - 4×s.clue_count)
分项2 材料完整分 mat   = EV_CHAIN?.case_score != null ? round(case_score×100) : null
分项3 整改闭环分 rect  = prog.total === 0 ? 100 : round(prog.done / prog.total × 100)
总分 = mat != null ? round(fee×0.5 + mat×0.3 + rect×0.2)
                   : round(fee×0.7 + rect×0.3)          // 材料分缺失时权重重分配
```

等级映射（颜色用现有 CSS 变量体系）：

| 总分 | 等级 | 等级语 | 颜色 |
|---|---|---|---|
| ≥90 | A | 飞检准备良好 | `var(--green)` |
| 75–89 | B | 少量风险待整改 | `#B07C1B`（琥珀） |
| 60–74 | C | 存在暴露风险 | `#D97706`（橙） |
| <60 | D | 高暴露 · 立即整改 | `var(--red)` |

硬性要求：
- **G0 干净件**（无 findings）在院端模式必须得 A 级，hero 主标题显示「体检通过 — 未发现暴露风险」。
- 整改登记提交成功后（`saveRectificationCard` 的成功回调里）**重算并刷新总分**——让用户看到"登记一条，分数涨一点"的正反馈。实现：提交成功后若 `MODE==='exam'` 调用重渲染总览页 hero 的轻量函数（只更新 hero 节点，不整页重渲）。
- 分数构成透明：等级徽章悬停弹出构成明细（复用 `.ch-pop` 的悬停弹层模式，见 `renderEvCaseHead`），内容逐字如下：

```
体检总分构成（演示口径 · 全部可溯源）
费用合规 {fee}/100 —— 每疑点 -12 · 每线索 -4
材料完整 {mat}/100 —— 即左栏「案卷完整度」金额加权分
整改闭环 {rect}/100 —— 已提交登记 {done}/{total} 条
加权 50% / 30% / 20%
```

### 1.3 视觉规格

- SVG 圆环：外径 92px，描边 8px，底环 `#E3F5EC`，进度环用等级色，`stroke-dasharray` 按分数比例，圆环中心两行——大号分数（`font-family:var(--font-num)`，34px/800）+ 小字「体检总分」。**禁止引入任何图表库**，手写 SVG。
- hero 布局：圆环居左，右侧三行：
  - 第一行：`<span class="rh-badge">院端体检报告</span>` + 等级徽章 `等级 {A/B/C/D} · {等级语}`
  - 第二行（主标题，沿用 `.rh-title`）：有风险时 `暴露 {suspected_count} 处风险点 · ¥{fmt(suspected_amount)} 待整改`；干净时 `体检通过 — 未发现暴露风险`
  - 第三行：三个分项小分 chip：`费用合规 {fee}` / `材料完整 {mat 或 "未评估"}` / `整改闭环 {rect}`，每个 chip 左侧一个 6px 圆点用对应分数的等级色
- 监管模式 hero **完全不走这个分支**，`reportHeroHTML(report, s, exam)` 内 `if (exam) return examHeroHTML(...)` 提前分叉。

### 1.4 验收

1. 院端跑肿瘤违规案卷 → 出圆环+等级 B/C/D，分项与公式一致（人工验算一条）。
2. 院端跑 G0 干净件 → A 级绿环，「体检通过」。
3. 切回监管模式重跑 → hero 是原版「已定位 N 条可回链疑点」，无任何分数元素。
4. 在整改 Tab 提交 1 条登记 → 回到体检报告页总分上升。

## P2 · 疑点卡「卡头语法」分叉（同一条 finding 两种读法）

改动集中在 `findingCard(f)`（L1856）。用现有全局 `VIEW_EXAM` 分叉。

### 2.1 院端卡头（VIEW_EXAM = true）

卡头徽章从左到右的**新顺序与内容**：

1. **三阶段 chip**（新增）：`examStageOf(f)` 返回 事前/事中/事后，渲染为：
   - `🟢 事前 · 开单可防`（class `stage-chip st-pre`）
   - `🟡 事中 · 结算前可拦`（class `stage-chip st-mid`）
   - `🔴 事后 · 需深查`（class `stage-chip st-post`）
   - chip 的 `title`：事前=`开单/上传前两库提醒即可拦住——建议接入事前提醒`；事中=`结算上传前自查可发现——避免被退回/拒付`；事后=`深读案卷才能发现——飞检重点，优先整改`
2. **责任科室 chip**（新增）：`🏥 {RECORD.front_page?.admit_dept || '医保办'}`（class `dept-chip`，title=`默认按入院科室归属，登记整改时可改责任人`）
3. 状态徽章：沿用现有 `疑点→风险点` 改写（L1861 已实现）
4. 规则名 + 金额（沿用）

院端卡头**隐藏**以下监管专属徽章：`tri-badge`（明确违规/可疑定性）、`violation_nature`（主观嫌疑）徽章——院端不做定性，只做整改。实现：这两个徽章渲染处加 `!VIEW_EXAM &&` 条件。

`examStageOf(f)`：**逐行照抄** `prototype/app/engine/three-stage.js` 的 `classifyStage(f)` 到 app.js（含正则），函数头加注释：
`// 与 engine/three-stage.js classifyStage 保持同步——改动必须双边同改`。

### 2.2 院端卡内：行内快速整改条（替换现有跳转 link）

改造 `renderExamRectification(f)`（L1684）：从一个"→ 去登记整改"link 升级为**行内快速整改条**（不替代整改 Tab 的完整表单，只做最常用两件事）：

```
[整改状态 ▾（5 项，同 rectificationCard 的 statuses）] [期限 date 输入] [💾 快速暂存] [完整登记 ↗]
已暂存后追加小字：✓ 已暂存 {status} · 期限 {deadline}
```

- 保存走既有 `/api/rectification` POST，payload 同 `saveRectificationCard` 但 `submitted:false`（快速条只做暂存，"提交登记+人工判断"仍必须去整改 Tab——判断对错是治理回流入口，不能简化掉）。
- 「完整登记 ↗」保留原跳转行为。
- 状态为 `已整改`/`已主动退回` 时，整张疑点卡加 class `f-rectified`，卡头右侧追加徽章 `✓ 已整改留痕`（绿色）。

### 2.3 监管卡头（VIEW_EXAM = false）

只做一处增强——**三要素齐备徽章**（呼应"引不出原文的疑点，不输出"）：

- 新增 `forensicReadyBadge(f)`：`f.evidence?.length && f.policy?.length && f.reasoning` 三者齐备时渲染 `⛓ 三要素齐 · 可举证`（class `forensic-badge ok`，印章感样式见 P5）；有缺时渲染 `⛓ 缺{缺失项中文名}`（class `forensic-badge warn`），缺失项名：证据定位/政策条款/推理过程。
- 插在金额徽章之前。院端不渲染此徽章。

### 2.4 验收

1. 院端跑违规案卷：每张卡可见 阶段chip+科室chip，无"明确违规/主观嫌疑"徽章；行内快速条可暂存，刷新后状态保持（RECT_MAP 从 `/api/rectification` 回读）。
2. 监管同案卷：卡头有定性徽章+三要素徽章，无阶段/科室 chip、无快速整改条。
3. 阶段分类抽 3 条与 `洞察→三阶段自查` 弹窗（走服务端同名逻辑）比对，分类一致。

## P3 · 左栏费用锚：院端换"整改工单"语义

改动集中在 `buildLineNature()`（L574）、`renderAnchorList()`（L690）、`index.html` L118-122 排序条。

### 3.1 数据：行 → findings 映射

`buildLineNature()` 现在只存每行疑点档；扩展为同时记录 `finding_ids`：`LINE_NATURE[line_no].finding_ids = [...]`（在它遍历 findings 关联行号的既有循环里顺手收集，不另写遍历）。新增辅助：

```
function lineRectStatus(line_no)
  → 取该行关联 findings 的 RECT_MAP 状态，聚合优先级：
    未整改(任一无记录或待整改) > 整改中 > 延期申请 > 已整改/已主动退回(全部) 
  → 返回 { key:'todo'|'doing'|'delay'|'done'|null, label }
  （null = 该行无关联 finding，即干净行）
```

### 3.2 排序条：院端追加「整改状态」档

`index.html` `#anchorSort` 内追加一个按钮：

```html
<button type="button" class="asort exam-only hidden" data-sort="rect">整改状态</button>
```

- `applyModeUI()` 中：`$$('.asort.exam-only').forEach(b => b.classList.toggle('hidden', !exam))`；且**院端模式下若已出报告，默认排序切到 `rect`**；切回监管时若当前是 `rect` 则回落 `risk`。
- `renderAnchorList()` 增加 `ANCHOR_SORT === 'rect'` 分支，分组渲染（复用 risk 分支的 `agroup-head` 结构）：

| 组序 | 组头文案 | 组头 class |
|---|---|---|
| 1 | `未整改 · {n} 行 · ¥{amt}` | `red` |
| 2 | `整改中 / 延期 · {n} 行 · ¥{amt}` | `amber` |
| 3 | `已整改留痕 · {n} 行 · ¥{amt}` | `green` |
| 4 | `无风险 · {n} 行 · ¥{amt}` | `green`（复用干净组样式） |

### 3.3 行徽标：院端换语义

`rowHTML` 中，`MODE==='exam'` 且该行有关联 finding 时，把风险徽标 `rbadge` 替换为整改徽标：

- 未整改：`待整改`（红描边）
- 整改中：`整改中`（琥珀）
- 延期申请：`延期`（琥珀虚线边）
- 已整改/已主动退回：`✓ 已整改`（绿实底）

监管模式行徽标维持现状（明确违规/可疑/干净）。完整度 `cbadge` 两模式都保留（它是中性的数据事实）。

### 3.4 右侧证据链面板（`evChainHTML` 尾部）

院端模式下，选中行若有关联 finding，在四层证据链下方追加一块：

```
🔧 整改建议：{examDisposal(f.disposal_suggestion) || '登记整改状态并留存说明材料'}
   当前状态：{整改状态 或 '未登记'} · [去登记 ↗]（跳对应 rect card，复用 P2 的跳转）
```

监管模式不出现此块。

### 3.5 验收

1. 院端出报告后左栏自动按整改状态分组；快速条暂存"整改中"后，左栏该行徽标实时变为「整改中」（暂存成功回调里触发 `renderAnchorList()`）。
2. 监管模式看不到「整改状态」排序按钮，行徽标为疑点档三色。

## P4 · 右栏常驻条 + 出口分叉

### 4.1 院端：整改进度常驻条

位置：`index.html` `#reportBody` 内、`.report-pager` 之前，新增 `<div id="modeStrip" class="hidden"></div>`（**在分页体外，翻页时常驻**）。

院端出报告后渲染（`renderReport` 尾部调用 `renderModeStrip()`）：

```
🩺 整改进度 ▓▓▓░░ {done}/{total} 条已登记 · 已消除 ¥{resolved}（来自 exam_diff，无则省略该段） 
· 最近期限 {最近的 deadline，格式 MM-DD，逾期红字「逾期 {n} 天」} 
[继续登记 →]（跳整改 Tab） [导出整改清单]
```

- 进度条：细 6px 圆角条，绿填充 `done/total`。
- `total===0`（干净件）时整条替换为：`🩺 本轮体检未发现待整改项 · 建议留存本次体检报告作为飞检前自查凭证 [导出体检报告]`。

### 4.2 监管：取证齐备常驻条

同一个 `#modeStrip`，监管模式渲染为：

```
⛓ 证据三要素齐备 {ready}/{n} 条（政策已核 {verified} 条） · 疑点涉及 ¥{amt} 
[生成举证包]（触发既有 #btnExport 的导出逻辑）
```

- `ready` = findings 中 evidence+policy+reasoning 齐备的条数；`verified` = policy 数组里 `verify_status` 含"已核"的条目数（与 `findingCard` 里 `vchip ok` 同判据）。

### 4.3 出口（导出）文案分叉

导出逻辑本体不动，只改产物标题/文件名/按钮文案：

| 位置 | 监管 | 院端 |
|---|---|---|
| 工具菜单 `#btnExport`（`applyModeUI` L246 已分叉，改词） | `导出举证包` | `导出整改清单` |
| 导出产物内标题行 | `鹰眼稽核举证包 · {案卷名}` | `鹰眼自查整改清单 · {案卷名} · 附整改承诺` |
| 院端导出尾部追加固定一行 | — | `本机构承诺：以上风险点已如实自查登记，整改留痕备查。——飞检前主动自查凭证` |

### 4.4 验收

翻到报告任意分页，常驻条都在；两模式常驻条内容/按钮完全不同；干净件院端出"未发现待整改项"版本。

## P5 · 视觉与微文案收尾（细节感）

### 5.1 CSS（`style.css`，全部收在既有 `body.mode-exam` 区块附近）

1. 院端圆角加大：`body.mode-exam{ --radius-card:16px }`，`.finding`、`.scard`、`.rect-card` 的 `border-radius` 改用 `var(--radius-card,12px)`（监管默认 12px 不变）。
2. 监管「印章」徽章：`.forensic-badge.ok{ border:1.5px solid var(--yy-ink,#0B2A4A); color:#0B2A4A; font-family:var(--font-num); letter-spacing:.08em; border-radius:4px; padding:2px 8px; background:transparent; transform:rotate(-1.5deg) }`——方角、微旋转、描边，卷宗印章感。院端一律不用此样式。
3. 阶段 chip：`st-pre` 绿底、`st-mid` 琥珀底、`st-post` 红底，统一 11px/圆角 6px/700 字重。
4. 已整改卡：`.finding.f-rectified{ opacity:.82; border-left:3px solid var(--green) }`。
5. 体检圆环等级色只用 §1.2 表中四色，禁止新增色值。

### 5.2 顶栏与空态微文案

| 元素 | 监管 | 院端 |

## §7 表述总表（词汇红线 · 执行时唯一措辞依据）

### 7.1 双方词汇表

| 概念 | 监管侧用词 | 院端用词 | 严禁混用说明 |
|---|---|---|---|
| 发现的问题 | 疑点 | 风险点 | 院端界面严禁出现"疑点/涉嫌" |
| 动作 | 稽核、取证、调阅、对质、举证 | 体检、自查、整改、留痕、补全 | 监管界面严禁出现"整改进度/主动退回建议" |
| 严重定性 | 明确违规 / 可疑（tri-badge） | （不定性，只分 事前/事中/事后） | 院端隐藏定性徽章 |
| 金额 | 疑点涉及 ¥ / 涉嫌金额 | 飞检暴露 ¥ / 待整改金额 | |
| 处置 | 处置建议（移交/责令退回…原文） | 自查整改建议（经 `examDisposal` 改写） | `examDisposal` 已有，勿绕过 |
| 终点 | 举证包 | 整改清单 + 体检报告 | |
| 重跑 | 重新稽核 | 复检对比 | |
| 材料请求 | 需调阅材料清单 | 建议补全材料 | 已有（L1859），保持 |

### 7.2 数字表述规范（细节感关键）

- 金额一律 `¥{fmt(n)}`，千分位；分数一律 `{n}/100`；条数一律 `{n} 条`；行数 `{n} 行`。
- 所有比率必须给分母：写 `3/7 条已登记`，不写 `43%`。
- 期限逾期必须红字并给天数：`逾期 3 天`，不写"已逾期"。
- 悬停 title 是细节感主战场：本方案每个新增徽章/chip 都规定了 title，**一个都不能省**。
