# 鹰眼 · 知识架构（KB Architecture v1.0）

> 解决的问题：**导入的规则与知识很多（KB1 ≈ 4583 条，其中两库 3893 条），但只是「原料堆」——没有二次整合与提炼，引擎消费方式停留在 ref→原文查字典。**
> 本文定义鹰眼自己的五层知识架构，以及「原料 → 可执行知识」的提炼纪律。
> 配套实现：`scripts/build-kb-operational-index.mjs` → 产出 `prototype/data/kb/kb_operational_index.json`。

---

## 一、五层架构（L0 原料 → L4 规则）

```
L0 原料层   raw/：邮箱附件 xlsx/pdf、官网爬虫、全书 PDF —— 只增不改，保留原件
     ↓ parser（xlsx-liangku / pdf-liangku / import-liangku-book-2025）
L1 规范层   kb1_policies.json：统一 schema（ref_id/doc_id/text/metadata），
            content_key 幂等去重，✅已核实 / ✅爬虫入库(待人工抽检) 双态标注
     ↓ build-kb-operational-index（二次整合：本文新增）
L2 模式族   把 20 个官方规则类归并为 ~9 个「检测模式族」——同族共享一个判定形状
L3 操作索引 kb_operational_index.json：药品/项目名 → 结构化约束（可被 checker 直接执行）
     ↓ 引擎消费
L4 规则层   rules.yaml 58 条（人工策划）+ 数据驱动 checker（吃 L3 索引，一个 checker 覆盖一族）
```

**核心思想**：官方两库按「发布批次」组织（第1批性别、第9批二线……），那是**发文视角**；
鹰眼按「检测模式」重组，那是**执行视角**。一个模式族 = 一种判定形状 = 一个数据驱动 checker 模板。

## 二、九个模式族（L2）

| 族 id | 判定形状 | 归并的官方类 | 条目量级 |
|---|---|---|---|
| `gender_limited` | 项目/药品 × 患者性别 ≠ 限定性别 | 药品区分性别使用 / 医疗服务项目区分性别使用（+全书：手术编码×性别、耗材×性别、诊断×性别） | ~1000 |
| `child_limited` | 药品/项目 × 患者年龄 > 儿童上限 | 药品儿童专用 / 药品限儿童使用 / 医疗服务项目儿童专用（+耗材限儿童/新生儿） | ~410 |
| `indication_limited` | 药品 × 诊断 ∉ 限定适应症 | 药品限适应症（12–14批） | ~510 |
| `second_line` | 用药 × 无一线无效/不耐受证据 | 药品限二线使用 | ~110 |
| `mutual_exclusive` | 项目A × 项目B 同时段收费 | 医疗服务项目重复收费（A×B 成对） | ~900 |
| `facility_level` | 药品 × 机构级别 < 限定级别 | 药品限医疗机构级别 | ~62 |
| `insurance_type` | 药品 × 险种 ≠ 工伤/生育 | 药品限工伤保险 / 限生育保险 | ~12 |
| `tcm_no_pay` | 饮片 ∈ 不予支付清单（单方/单复方） | 中药饮片单方 / 单复方均不予支付 | ~153 |
| `usage_limit` | 疗程/频次/周期/互联网范围（暂存待接） | 限支付疗程 / 限定频次 / 周期超频次 / 超互联网医院 | ~130 |

每条索引项都带 `refs[]` 指回 L1 的 `ref_id` —— **提炼不丢溯源**，报告引证仍能点到官方原文与核验状态。

## 三、整合纪律（做过什么二次处理）

1. **类名规范化**：去中文引号、去「第N批…明细」前缀 → 27 个脏类名收敛为 20 个规范类，再归 9 族。
2. **跨来源去重合并**：同 (族, 名称) 的批次 xlsx 条目与全书 PDF 条目合并（~399 组重复）——医保编码取并集、限定条件取最长文本、refs 全保留。
3. **条件结构化**：从 `detect_logic` / `payment_basis` 抽出可执行字段：
   - `limit_sex`（男/女）← "男与限定性别不符"
   - `age_min/age_max` ← "限3-12岁…"（儿童族默认上限 14）
   - `level_min` ← "限二级及以上医疗机构"
   - `insurance` ← 工伤/生育
   - `pair.a / pair.b` ← "急诊监护费×急诊诊查费"
   - `tcm_mode` ← single（单方不予支付）/ both（单复方均不予支付）
4. **名称索引**：`constraints` 以规范化名称为键；查询用「医嘱名 contains 索引名」匹配（饮片用精确匹配防止「阿胶」误伤「复方阿胶浆」）。

## 五、重建时机

```bash
# 任何两库导入/更新后重跑（幂等，秒级）：
node scripts/build-kb-operational-index.mjs
# 然后照常回归：
bash yhf/run.sh --strict
```

产物 `kb_operational_index.json` 是**派生文件**（可随时从 L1 重建），不手改；手工核实/修正只发生在 L1 的 `verify_status` 与 L4 的 `rules.yaml`。
