# 鹰眼 · 监管级引用闭环（处方式政策标注）· 执行方案

> 交接对象：执行 AI / 全栈工程。读完本文即可动手，不需要再问需求方。
> 日期：2026-07-04 · 基于当前 main 工作区代码。
> 一句话目标：把「每条结论像开处方一样标注两库批次-条目-原文出处，两层锚点可点穿，引用不出条目 ID 的疑点自动降级转人工」从 pitch 话术变成**主界面每条疑点上可点、可验、可导出的事实**。

## 0. 背景 · 现状 · 总原则

### 0.1 问题诊断（先读，防止做错方向）

评委的问题是「这个结论我敢不敢签字」。签字感来自三件事：**引用具体到条目、条目可点穿到原文、引不出就明说转人工**。当前仓库的真实状态：

| 环节 | 现状 | 差距 |
|---|---|---|
| KB1 数据 | 11,027 条，每条有 `ref_id/locator/verify_status`，10,683 条带 `metadata.attachment`（官方 PDF/xlsx 文件名），7,937 条带 `source_url`，10,337 条带 `metadata.batch`（格式脏） | 页码字段**全库为 0**；批次值需归一化 |
| 条目索引 | `buildPolicyMaps()`（`prototype/app/kb/retrieval.js` L37）已构建 `policyMeta`（ref_id → 完整条目） | **没接进稽核 ctx**，引擎拿不到 |
| 引擎输出 | `audit-engine.js` L1181 把 policy 压成 `{ref, text, verify_status}` 三字段 | 批次/文号/locator/官方链接在此被剥掉 |
| 引用兜底 | `lookupPolicy()` L1237：查不到原文时输出占位符 `（KB1/KB2 取原文，引用ID: xxx）` | 占位符照常展示，**不降级不告警** |
| 合规门禁 | `compliance-gate.js` C-002：全部条款不在库才降级线索 | 是文本存在性检查，不是「条目可解析」硬校验；降级后 UI 无醒目标注 |
| 转人工硬校验 | 仅 `tri-persona.js` Q9（L194-202）：合议裁定引用不出弹药库 ID → 强制「证据不足转人工」 | **只覆盖按需触发的合议路径**，不覆盖默认稽核主路径 |
| UI | `findingCard()` 的 `polHtml`：`pref` 是不可点的 `<span>` | 第二层锚点（条款→KB 条目→官方原文）**不存在**；第一层（证据→费用行，`jumpToLoc`）已可用 |
| 导出 | `evidence-package.js` 举证包只列 `ref + text` | 举证包没有处方级引用，签字场景恰恰在导出物上 |

### 0.2 设计总原则（执行时时刻对照）

1. **诚实红线（最重要）**：库里没有的信息**绝不显示**。页码字段现库存为 0——处方行用「知识点 #序号」（`metadata.row_seq`，官方明细表里的真实序号）替代页码，**禁止编造页码**。页码补录列为 P6 数据工程，不做完就不展示。
2. **单一事实源**：条目富信息只从 `policyMeta`（源头 `kb1_policies.json` / `kb2_clinical.json`）解析，不在前端、不在规则文件里另存一份副本。
3. **主路径对齐 Q9**：确定性引擎主路径的硬校验语义与 `tri-persona.js` Q9 一致——「引用不出可解析条目 → 不出机器定性」。但主路径动作是**疑点降级线索 + 显式转人工标注**（保守，不静默丢弃，保可演示性）。
4. **不破坏既有门禁**：G0 零误报红线、三要素门禁、C-002 行为全部保留；新校验只会**更严**（把原本冒充疑点的弱引用压下去），不会放松。改完必跑 `bash yhf/run.sh --strict`。
5. **两层锚点定义**（写进 UI 文案的概念，先统一）：
   - 第一层（临床层）：疑点 → 病历/费用行原文，已有（`jumpToLoc`），本方案不动；
   - 第二层（政策层）：疑点 → KB 条目 ID → 条目详情（原文全文/批次/官方链接/核验状态），本方案新建。

## P1 · 引用解析层：citation-resolver + policyMeta 接线 + 条目 API

### 1.1 新模块 `prototype/app/engine/citation-resolver.js`

单一职责：`ref 字符串 → 处方级引用对象`。接口：

```
buildCitationIndex(kb1, kb2) → index        // 启动时一次，ref_id/doc_id 双键 + 排序数组供前缀匹配
resolveCitation(ref, index)  → entry|null   // 精确命中 → 前缀互含（照抄 foundation.js resolveRef 语义）→ null
formatCitation(entry)        → citation     // entry → 展示对象（见 1.2）
normalizeBatch(entry)        → string|null  // 批次归一化（见 1.3）
```

**resolve 顺序**：`index.byRefId[ref]` 精确 → `foundation.resolveRef` 同款前缀互含扫描 → null。问题清单合成 key（`KB1-问题清单…`，只在 policyTexts、不在 policyMeta）resolve 不到 entry 时，**不算失败**：citation 降级为 `{ synthetic: true, note: '问题清单聚合条目（行业口径）' }`，存在性以 `policyTexts[ref]` 为准——这是现实数据形态，硬校验时同等对待（见 P2）。

### 1.2 citation 展示对象（处方行的数据契约）

```json
{
  "ref": "KB1-两库2025-医疗服务项目重复收费-急诊监护费-急诊诊查费-1",
  "resolved": true,
  "source_kind": "两库知识点 | 法规条款 | 目录 | 问题清单 | 临床指南",
  "batch_label": "国家两库·第七批",
  "rule_category": "医疗服务项目重复收费",
  "item_name": "急诊监护费×急诊诊查费",
  "seq": 1,
  "codes": ["001103000010000", "001102000030000"],
  "doc_name": "第七批「医疗服务项目重复收费」规则对应知识点明细.xlsx",
  "doc_no": "国务院令第735号（仅法规类有）",
  "locator": "第三十八条（法规类）/ 规则类目名（两库类）",
  "authority": "国家医疗保障局",
  "effective_from": "2025-07-22",
  "source_url": "https://www.nhsa.gov.cn/art/2025/7/22/art_109_17342.html",
  "attachment_name": "第二批「手术项目未按规定折价收费」规则对应知识点明细.pdf",
  "verify_status": "✅爬虫入库(待人工抽检)"
}
```

字段规则（诚实红线的落点）：

- `source_kind` 由 `doc_id`/`layer` 推导：`KB1-条例*`→法规条款；`KB1-两库*`→两库知识点；`layer==='目录'`→目录；`KB1-问题清单*`→问题清单；KB2→临床指南。
- `attachment_name` 只取 `path.basename(metadata.attachment)`，**绝不输出本地绝对路径**。
- `source_url` 为 `book-2025-pdf` 这类非 URL 占位值时置 null（前端不渲染链接）。
- 任何字段缺失就是 null，前端跳过该段，**不填「暂无」不填猜测值**。

### 1.3 批次归一化 `normalizeBatch(entry)`

现库 `metadata.batch` 有 33 种脏值（`第二批` / `第七批-医疗服务项目重复收费-规` / `2025全书` / `国家医保局关于公开发布第十五批智`…）。归一化规则（按序尝试，全落空返回 null）：

1. `metadata.batch` 或 `doc_name` 中匹配 `/第([一二三四五六七八九十]{1,3})批/` → `国家两库·第{X}批`；
2. `metadata.batch === '2025全书'` 或 `crawl_source === 'liangku-book-2025'` → `国家两库·2025年版`；
3. `doc_id === 'KB1-条例'` → `《医疗保障基金使用监督管理条例》`（此类展示文号+locator，不展示批次）；
4. 其余 → null（处方行省略批次段）。

**归一化只在展示层做，不回写 KB 文件**——KB json 是爬虫产物，回写会被下次爬虫覆盖。

### 1.4 policyMeta 接线（三处，都是加一行级改动）

1. `server.js` 启动段（L297-315）：`DB.policyMeta = buildPolicyMetaFromKb(kb1, kb2)`，并建 `DB.citationIndex = buildCitationIndex(kb1, kb2)`。**注意 Supabase live 模式**：`loadFromSupabase()` 不返回 policyMeta，所以 citationIndex 必须始终从本地 JSON 构建（L297 的 `kb1` 变量就在手边），live 模式下照用——条目富信息展示允许比 live 文本晚一个爬虫周期，这是可接受的演示口径。
2. `server.js` `auditContextForRecord()`（L163）：ctx 增加 `policyMeta: DB.policyMeta, citationIndex: DB.citationIndex`。
3. `audit-pipeline.js` `buildEnrichedContext()`（L33）：同样透传 `policyMeta/citationIndex`（从 `policyMapsRaw.policyMeta` 取，缺则由调用方兜底）。`audit-engine.js` L1703、L1872 的 ctx 构建处把 `options.policyMeta/citationIndex` 收进 ctx。

### 1.5 新 API `GET /api/kb/entry?ref=<ref_id>`

挂在 `server.js` L1250 `/api/kb` 路由区旁边。行为：

- `resolveCitation(ref, DB.citationIndex)` 命中 → 返回 `{ entry:{...完整条目, metadata.attachment 替换为 basename}, citation: formatCitation(entry) }`；
- 未命中但 `DB.policyTexts[ref]` 存在 → `{ entry:null, synthetic:true, text: DB.policyTexts[ref], citation:{ref, resolved:false, note:'聚合/合成条目，无独立官方条目'} }`；
- 都没有 → 404 `{ error:'引用不可解析', ref }`。
- **禁止**返回全库（现有 `/api/kb` 返回 11k 条 ~ 几十 MB，前端点条款绝不能走它）。

### 1.6 验收

1. `node -e` 冒烟：对 `KB1-条例-第38条`、一条第七批 xlsx 条目、一条 2025 全书条目、一个问题清单合成 key、一个乱写的 ref 分别调 `resolveCitation`+`formatCitation`，检查 5 种输出形态与 1.2 契约一致。
2. `curl 'http://localhost:3700/api/kb/entry?ref=KB1-条例-第38条'` 返回含 `citation.locator === '第三十八条'`、`doc_no === '国务院令第735号'`；响应体 < 20KB。
3. 返回体里 grep 不到 `/Users/`（本地路径不外泄）。

## P2 · 引擎输出扩展 + 主路径「引不出条目 → 转人工」硬校验

### 2.1 finding.policy 结构扩展（`audit-engine.js` L1181）

现在：

```js
policy: (rule.policy_basis || []).map(ref => ({ ref, text: lookupPolicy(ctx, ref), verify_status: policyVerifyLabel(ctx, ref) }))
```

改为每个 policy 对象追加 `citation` 字段（`resolveCitation` + `formatCitation` 的产物，resolve 失败时 `citation: { ref, resolved:false }`）。**向后兼容**：`ref/text/verify_status` 三个旧字段原样保留，所有旧消费方（priority.js、插件、导出、eval）不受影响。

`lookupPolicy()`（L1237）占位符行为保留（LLM 上下文还依赖它），但占位文案改为更诚实的 `（该引用在知识库中无原文——已标记引用未穿透）`。

LLM 路径（`llm-audit.js` L112 回填 verify_status 处）做同样的 citation 回填——LLM 只允许引用传入 KB 的 ref，所以 resolve 逻辑完全复用。

### 2.2 主路径硬校验：C-006 + finding 级 `citation_integrity`

在 `compliance-gate.js` 的疑点分支追加（放在 C-002 之后）：

```
对每条 policy 判定 resolvable = resolveCitation 命中 || policyTexts[ref] 存在
finding.citation_integrity = { total, resolved, unresolved_refs:[...] }
若 status==='疑点' 且 resolved === 0：
  status → '线索'
  finding.needs_human = true
  add('C-006','downgrade_manual','引用不出可解析条目ID，按 Q9 同款硬约束降级并转人工复核')
```

语义要点：

- 判定口径与 C-002 的差别：C-002 看「文本在不在库」，C-006 看「条目能不能解析成引用」并且**留下转人工的显式标记**。两者都保留——C-002 是降级，C-006 是降级+转人工声明+完整性台账。
- `partial`（部分条款解析、部分失败）**不降级**，只记录 `citation_integrity`，UI 上给黄色提示（见 P3）——一条疑点常引「主条款+辅助指导」，主条款硬即成立，这是 C-002 修 bug 时确立过的语义（见该文件注释），不许回退。
- **报表聚合**：`audit-engine.js` 组装 `report_meta.summary` 处追加 `citation:{ findings_fully_cited, findings_partial, findings_manual, refs_total, refs_resolved }`，供 UI 总览条和 G8 门禁共用一个口径。

### 2.3 回归预期（执行者必须先跑基线再改）

改动前先 `bash yhf/run.sh` 记录各 gate 数字。C-006 理论上不应改变任何演示案卷的疑点数——`rules.json` 的 `policy_basis` 都是在库 ref。如果跑完发现某案卷疑点被 C-006 降了，**那是发现了真实的坏引用**：修数据（rules.json 的 ref 或 KB 补条目），不是放松门禁。G0 干净件不受影响（干净件本来无疑点）。

### 2.4 验收

1. 单测式冒烟：构造一条 policy 全为乱 ref 的假疑点过 `applyComplianceGate` → status 变线索、`needs_human===true`、compliance_flags 含 C-006。
2. 全量案卷跑批（`/api/audit/batch` 或 yhf harness），`summary.citation.findings_manual === 0`（演示数据应全绿）。
3. `bash yhf/run.sh --strict` 全过。

## P3 · UI：处方行 + 第二层锚点点穿 + 转人工印章

### 3.1 处方行（`findingCard()` 的 `polHtml` 重写，app.js 约 L2138）

每条政策引用从「ref + 核验 chip + 一段文字」升级为**两行结构**：

**第一行 · 处方行**（新，`class="policy-rx"`，等宽数字字体 `var(--font-num)`）：

```
📜 {batch_label} · {rule_category}「{item_name}」· 知识点 #{seq}{codes 有则 · 代码 {codes[0]}}{effective_from 有则 · 生效 {effective_from}} {verify chip} [原文 ↗]
```

真实数据示例（执行者可用第七批条目自验）：

```
📜 国家两库·第七批 · 医疗服务项目重复收费「急诊监护费×急诊诊查费」· 知识点 #1 · 代码 001103000010000 {✅待抽检} [原文 ↗]
```

法规类（条例）变体：

```
📜 《医疗保障基金使用监督管理条例》国务院令第735号 · 第三十八条第(二)项 · 自2021-05-01施行 {✅已核验} [原文 ↗]
```

分段规则：citation 字段为 null 的段直接省略（诚实红线）；`resolved:false` 时整条处方行替换为警示行：`⚠ 引用 {ref} 未穿透到知识库条目 — 已转人工复核`（红字，class `policy-rx unresolved`）。

**第二行 · 原文摘录**：保留现有 `p.text` 展示（截断 240 字符 + 「…」），样式不变。

### 3.2 第二层锚点点穿：`showPolicyEntry(ref)` 弹窗

处方行整行与 `[原文 ↗]` 按钮都可点，调 `window.showPolicyEntry(ref)`（新增，模式参照现有 `showRuleDetail`）：

1. `fetch('/api/kb/entry?ref='+encodeURIComponent(ref))`；
2. `openModal('📜 官方条目 · ' + ref, …)` 渲染：
   - 头部：处方行完整重现 + `verify_status` 原文（三态 chip 样式沿用 `.vchip`）；
   - **原文全文**（不截断，`white-space:pre-wrap`）；
   - 溯源块（kv 表）：来源文件（`attachment_name`）、官方链接（`source_url` 有效时渲染 `<a target="_blank">` 官网原文 ↗）、发布机关、生效日、检出逻辑（`metadata.detect_logic` 有则展示）、关联规则（`linked_rules` 渲染成 `ruleLink` chip，点击回跳规则详情——**这就把 规则↔条目 双向打通了**）；
   - 底部一行小字：`条目取自本地知识库快照（{kb_meta.verified_date} 核验口径）· 绝不凭记忆生成条款`；
   - `synthetic:true` 时如实显示「问题清单聚合条目 · 无独立官方条目，供旁证参考」。
3. 404 时弹窗显示 `⚠ 引用不可解析——本条已按规程转人工复核`，与 C-006 文案闭环。

同样把 `showRuleDetail`（app.js L97）里 `policy_basis` 的纯文本 `<li>`（L69）改成可点的 ref 链接，调同一个 `showPolicyEntry`——规则目录侧也获得二层穿透。

### 3.3 转人工与完整性的可视化（三处）

1. **卡头印章**：沿用双模式方案 P2.3 的 `forensic-badge` 印章体系，升级判据——三要素齐 **且** `citation_integrity.resolved > 0` 才盖 `⛓ 三要素齐 · 引用可穿透`；`needs_human` 时盖红章 `⚠ 引用未穿透 · 转人工`（`forensic-badge warn`，同款印章样式）。partial 时章后缀 `（{resolved}/{total} 条款可穿透）`。
2. **总览统计**：`evidenceActionsHTML()`（约 L1421）现有的「三要素齐备 ready/n · 政策已核 verified 条」统计行扩为：`⛓ 三要素齐 {ready}/{n} · 引用可穿透 {findings_fully_cited}/{n} · 待人工 {findings_manual} 条`，数字直接读 `report_meta.summary.citation`（2.2 的口径，前端不重算）。

### 3.4 院端体检模式的措辞分叉（对齐双模式方案 §7 词汇表）

## §7 表述规范（处方行 grammar · 唯一措辞依据）

### 7.1 处方行分段语法（按 source_kind）

| source_kind | 模板 | 缺段处理 |
|---|---|---|
| 两库知识点 | `📜 {batch_label} · {rule_category}「{item_name}」· 知识点 #{seq} · 代码 {code} · 生效 {date}` | 任一段缺即整段省略，`·` 分隔符随之收缩 |
| 法规条款 | `📜 《{doc_name}》{doc_no} · {locator} · 自{effective_from}施行` | doc_no 缺则省略 |
| 目录 | `📜 {doc_name} · {locator} · {region}` | region='全国' 时省略 |
| 问题清单 | `📜 行业问题清单（聚合口径）· {locator}` | 固定加「聚合口径」，不冒充官方条目 |
| 临床指南(KB2) | `📖 {doc_name} · {locator}` | 图标用 📖 与政策区分 |

### 7.2 词汇红线

- 「知识点 #N」指官方明细表序号（`row_seq`），**禁止**写成「第 N 页」「p.N」（P6 前）。
- 核验三态措辞固定：`✅已核验`（人工）/ `🕒爬虫入库·待人工抽检` / `⚠待核验逐字原文`——沿用 `policyVerifyLabel` 现有输出，UI 不得改写这三个词。
- 转人工措辞固定：监管侧 `引用未穿透 · 转人工复核`；院端 `建议人工核对依据`；禁止出现「引用失败」「查无此条」等含糊词。
- 弹窗底注固定句：`条目取自本地知识库快照 · 绝不凭记忆生成条款`。
- 所有悬停 title：处方行 title = ref 全文；红章 title = `按 Q9 同款硬约束：引不出可解析条目ID的结论不出机器定性`。

### 7.3 数字规范

比率带分母（`引用可穿透 6/7`）；条目数写 `{n} 条`；ref 一律 `<code>` 等宽体；日期 `YYYY-MM-DD`。
