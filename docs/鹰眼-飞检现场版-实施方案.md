# 鹰眼 · 飞检现场版 — 实施方案 v1.0

> 定位:监管侧产品线的第三视图"现场模式"。把数据端疑点延伸为现场实证闭环,补齐官方"非现场监管与现场监管紧密结合"的后半句。
> 铁律:**这是既有系统的延伸,不是新系统。** 全部能力以插件(技能模块)形式挂载,复用优先于新建。

---

## 一、与既有设计的结合点(逐一对位,施工时照此接线)

| 既有资产 | 飞检现场版如何复用 | 严禁行为 |
|---|---|---|
| **批量筛查 + 优先队列 + 三档分类** | 行前任务生成器的唯一输入源:从队列勾选疑点(默认全部"可疑"档+高金额"明确违规"档)生成现场任务 | 不许另建"飞检疑点"数据结构 |
| **evidence_links 物化表 + 证据链视图** | 现场取证写入同一张表,新增 `layer=4(现场实证层)`、`source="onsite"`;证据链视图在原四层下方长出第五排节点 | 不许建平行的"现场证据表" |
| **规则 DSL 元数据**(GZ编码/三态文案/时效区间/处置建议/属地) | 任务卡的"核查依据"字段直接引用规则元数据;认定依据按第6号令第21条 = 法规 + **被检统筹区属地政策** + 结算日期时点法(字段现成) | 不许在任务卡里手写依据文本 |
| **三人格合议模块** | 复用为"争议研判":被检对象陈述申辩(第18条)录入后,合议给研判建议,输出挂"集体决策记录"——只换入口和输出模板,不改合议内核 | 不许为飞检另训一套人格 |
| **沉淀闭环(候选池+回放diff)** | 现场核实"不属实"的排除理由自动写入对应规则的误报统计,达阈值触发规则调参建议——飞检成为规则库最高质量的反馈源 | 不许让排除结果只留在飞检模块内 |
| **一键报告生成器(E3)** | 新增两个模板:当日碰头会小结、飞检报告骨架(第22条)+移交材料清单——同一报告引擎,只加模板 | 不许新写报告渲染逻辑 |
| **院端自查报告(院端产品线)** | 自查比对站:医院自查报告 vs 引擎全量扫描 diff,漏报项自动置顶为现场重点(2024年起自查自纠情况列为飞检重点内容) | — |
| **桌面哨兵插件架构(F2)** | 现场速查、询问提纲生成注册为哨兵技能(热键调用);飞检地图整体是工作台的一个视图插件 | 不许做成独立应用 |
| **调用链留痕面板(G3)** | 现场每次 tick/取证/改档写入同一留痕流(时间戳+操作人+设备),满足第17条"记录及时准确完整有效" | 不许单独造日志系统 |
| **schema校验包装器 + 四级降级梯子(C1)** | 智能体四技能(速查/提纲/小结/方案)全部走该包装器调内网千问 | 不许裸调 LLM |

## 二、插件化架构(按人员角色切分,各取所需)

**原则:一个内核 + 一组技能插件 + 三种载体。** 每个技能是独立模块,带 SKILL.md 式清单(名称、角色可见性、输入输出、依赖资产、降级路径),注册进工作台/哨兵后按角色显隐。

### 技能清单

| 技能插件 | 服务角色 | 载体 | 优先级 |
|---|---|---|---|
| S1 任务生成器 | 组长/后方 | 工作台"现场模式"视图 | P0 |
| S2 科室节点地图+四组动线 | 组长(全局)/组员(本组线) | 工作台视图,响应式(现场用平板/手机浏览器) | P0 |
| S3 任务卡 tick+取证挂载 | 各小组成员 | 工作台视图 | P0 |
| S4 回填改档 | 数据组 | 自动(引擎内) | P0 |
| S5 当日碰头会小结 | 组长 | 报告引擎新模板 | P0 |
| S6 自查比对站 | 组长/政策组 | 工作台视图 | P1(杀手锏,半天) |
| S7 询问提纲生成 | 医疗组 | 哨兵热键技能 + 任务卡内按钮 | P1 |
| S8 现场速查(依据/属地政策/判例) | 全员 | 哨兵热键技能 | P1 |
| S9 行前实施方案起草(第14条) | 组长 | 报告引擎模板 | 预览(入口+示例产物) |
| S10 争议研判(合议复用) | 组长/集体决策 | 合议模块新入口 | 预览 |

### 角色 × 视图矩阵

- **组长**:地图全局进度、四组动线、今日证实金额累计、S5/S6/S9;
- **医疗/财务/信息/政策组员**:本组动线、任务卡、tick+取证、S7/S8;
- **数据组(后方)**:回填审核、疑点档流转确认、误报统计;
- **被检地配合人员**:无账号——本产品单向服务检查方,不给被检方任何视图(保密纪律,第9条)。

## 三、数据模型(全部为增量,不动既有表)

```
inspection_plan     { plan_id, org_id, period, teams[], status, created_from: "queue_selection" }
inspection_task     { task_id, plan_id, source_finding_id(疑点ID,外键→既有疑点表),
                      station_id, team, type: 设备存在性|进销存盘点|病历调阅|人员询问|资质核验|数据复核,
                      basis_ref(规则DSL元数据引用), evidence_requirements[],
                      compliance_flags: {need_two_officers: true, need_signed_transcript: bool},
                      state: 待分配|已分配|进行中|已核|已回填,
                      verify_result: 属实|不属实|存疑, verify_reason }
inspection_station  { station_id, name(科室/药库/设备科/病案室/财务科/信息科/医务科), tasks[] }
onsite_evidence     → 写入 evidence_links(layer=4, source="onsite",
                      payload: {photo_ref|doc_no|transcript_summary, ts, operator, officers:[..]})
daily_brief         → 报告引擎产物,非新表
```

**疑点档流转规则(S4 内核,一条都不能多)**:
- 可疑 + 属实 → 明确违规(坐实),处置建议按规则元数据带出;
- 可疑 + 不属实 → 排除,`verify_reason` 必填 → 写入规则误报统计;
- 任意 + 存疑 → 保持原档,打"需补证"标记,任务可克隆再派;
- 明确违规 + 属实 → 维持,现场证据加固证据链。

### 任务生成模板(三类必做,对应现场三大动作)

1. **设备存在性核查**:触发条件=收费项目属大型设备/器械集合且收费频次异常 → 任务:到 X 科室核实设备在位,取证=铭牌与序列号照片+采购发票编号,交叉=收费期间设备是否在院;
2. **进销存盘点**:触发条件=药品/耗材收费量疑点 → 任务:药库调出入库台账,取证=台账复印盖章+差量计算表,交叉=收费量 vs 出库量;
3. **病历调阅+人员询问**:触发条件=记录与收费不符类疑点 → 任务:病案室调原始病历+询问经治医师,取证=询问笔录(逐页签字/捺印提示),**S7 按手术/医嘱记录自动生成追问提纲**(如"请复述该台手术的入路与植入物数量")。

## 四、程序合规护栏(第6号令 UI 化,不是文档化)

- 任务卡常驻红字:`need_two_officers=true` 的任务显示"本任务需至少 2 名持执法证件人员参加"(第17条),tick 时要求填两名操作人;
- 询问类任务 tick 前弹确认:"询问笔录已由被询问人逐页签字/捺印"(第17条);
- 所有取证自动打时间戳+操作人,进 G3 留痕流("记录及时、准确、完整、有效");
- 任务卡"认定依据"栏强制显示:法规条款 + 被检统筹区属地政策 + 规则生效区间(第21条);
- 结论前反馈:S10 争议研判入口挂在"检查结论"页,提示"作出结论前应向被检查对象反馈,听取陈述申辩"(第18条)。
