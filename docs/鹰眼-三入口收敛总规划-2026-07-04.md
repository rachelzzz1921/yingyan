> **日期**：2026-07-04 · 赛前收敛版
> **定位**：把「已经做完的很多东西」收敛成**三个能独立讲、能自己走**的小故事。本文是执行总纲：动线怎么定、链路怎么修、文案怎么改、插件放在哪一步、什么降级收纳。
> **依据**：SSOT v2.0（口径唯一权威）、游园会 Pitch 脚本 v1.0（三记忆点）、工作总清单-核对版（能力底账）、现状梳理 briefing（战略层收敛已定：院端首楔子 + 监管标杆）。本文不改战略，只把战略落到 demo 的动线与文案层。

## 0 · 总判断（一句话）

**页面能力是齐的，但整个站现在是「讲解员的提词系统」，还不是「用户的产品」。**

三类问题，对应三个工作包：

1. **文案是备赛视角**：首页三张入口卡的正文写的是拦评委的话术（"老师是医院的？""您有三分钟，我从头走一遍？"），页面上挂着「PITCH 节奏·见人下菜」「评委常问·速答口径」「评委走时脑子里剩这三句」这些导演视角区块，界面里还散落 G0/YHF/RAG/LLM/供应商名/内部人名。→ **工作包 B**
2. **入口有了、回路没有**：三入口分岔清晰，但走进去之后——golden 走完出不来、priority 打开案卷回不去队列、role 上下文跨页丢失、覆盖地图只能回看板。→ **工作包 A**
3. **插件线最丰富、位置最模糊**：20+ 个组件能力都在，但没有按「用户一天的动线」排位；精做/预览/生产形态混在一格里；一键连播只覆盖约 60% 的工具。→ **工作包 C**

另有一批「不属于任何故事」的页面需要收纳降级。→ **工作包 D**

## 1 · 三个故事的目标形态（收敛的靶子）

三入口 = 三个可独立讲完的小故事。每个故事一句话、一个主角、一条不出戏的动线。**判断一切去留的标准：这个元素帮哪个故事推进了一步？一步都不推进的，收纳或删。**

### 故事① 黄金闭环 —— "一份案卷是怎么查清楚的"
- **主角**：稽核员面前的一份案卷。**记忆点**：每条结论都有出处；缺证据本身就是疑点。
- **动线**（已实现，7 步解锁递进）：`/golden.html` 选案卷 → 秒级初筛 → 三要素证据链 → 人工采纳/驳回 → 治理沉淀 → 整改复跑 diff → 退回测算 → 门禁验收。
- **现存断点**：走完第 7 步只有"可继续查看工作台/插件"，**没有回首页、没有换一份案卷重走**——演示完即被困住（这是"跳转不顺"最典型的一处）。
- **收敛后**：终点卡三个出口：`再走一遍（换案卷）` / `进完整平台（带当前案卷）` / `返回首页`。

### 故事② 完整平台 —— "每天一万多条，是怎么管过来的"
- **主角**：监管侧是基金监管科审核员；院端是医保办/医务科。同一引擎，两个前台。
- **监管动线**：`/priority.html` 批量分诊排序 →「在工作台打开」`/index.html?role=audit&case=x` 单案取证 → 报告/举证包 →（采纳驳回反哺规则沉淀）。
- **院端动线**：`/index.html?role=hospital` 自查体检 → 登记整改 → 复检对比 → 退回测算（与 golden 第⑤⑥步同源）。
- **现存断点**：从队列打开案卷后**回不去队列**（顶栏只有"⌂ 切换工作台"回 home）；audit↔hospital 切换丢当前案卷；priority 页自身不读 role，直接访问时院端口径失效。
- **收敛后**：队列 → 工作台 → 回队列成环；role 全程跟人走。

### 故事③ 插件线 —— "不动你的系统，装上就能用"
- **主角**：院端是医生/编码员/结算员三个岗位；监管是拿着 Excel 和文件夹的一线人员。**记忆点**：装上就能用、数据不出机。
- **院端动线（事前-事中）**：`mockhis` 医生开单被拦 → `coder-station` 编码降编 → `settle-station` 结算整单自查 → `plugin-dashboard` 看板汇总 → 未遵从单流向 `priority` 预警台账、监管核实后回执写回看板（**这是插件故事的收口，也是三故事互相咬合的点**）。
- **监管动线（事后小工具）**：`regulator-triage` 分诊 → `regulator-threshold` 阈值试算 → `regulator-fieldkit` 行装包 →（`onsite` 现场闭环）。
- **现存问题**：两条动线混在一页平铺；一键连播只串监管四步；追溯码/判例/数据体检三个预览工具没有入场口也没标"预览"；浏览器扩展和桌面哨兵在动线上没位置。
- **收敛后**：plugins.html 按"院端一天 / 监管一天"两条故事线重排，工具三级标注（精做/预览/生产形态），连播拆两条链。

### 插件到底在哪一步？（回答你点名的问题）

插件不是第三个"并列产品"，它是**同一引擎在业务时间轴上的前移**。一句话定位可以直接写进页面：

> **平台管"事后查清楚"，插件管"事前不发生"。** 医生开单那一刻拦下的每一单（插件），都是稽核员案头少掉的一条疑点（平台）；拦不住、对方坚持提交的，带着理由自动流进监管预警台账（priority）——插件故事的终点，就是平台故事的起点。

按这个定位，三入口的讲述顺序天然成立：①先看一份案卷怎么查（能力证明）→ ②再看一万条怎么管（规模化）→ ③最后看怎么让问题根本到不了案头（前移，商业楔子）。

## 4 · 工作包 C：插件线收敛

### C1 · plugins.html 按两条故事线重排

现状是"监管一堆卡 + 院端一堆卡"平铺。改成两条**带时序**的动线条（保留现有引擎脊柱区）：

```
院端一天（事前-事中）：
  医生开单被拦(mockhis) → 编码校验(coder) → 结算自查(settle)
  → 运营看板(plugin-dashboard) → 坚持提交的单流向监管预警(priority#预警台账)
监管一天（事后）：
  Excel 分诊(triage) → 阈值试算(threshold) → 行装包(fieldkit) → 现场闭环(onsite)
  支线随用随查：追溯码 / 判例 / 数据体检
```

每张工具卡补两行小字：**「输入从哪来 → 输出到哪去」**（例：分诊的输出是试算的输入；行装包的输出带进现场）。这一行小字就是"插件属于动线哪一步"的答案，写在卡上，不用靠讲。

### C2 · 一键连播拆两条

现在只有监管链（triage→threshold→fieldkit+申诉加映）。补院端链：`mockhis?scenario=1 → coder → settle → plugin-dashboard`（场景播放器 TK.scenario 已是现成范式，院端三站补 scenario 剧本即可，剧本素材直接用 docs/插件三角色闭环-演示脚本.md 的 S1/C1/结算场景）。申诉批阅保持"加映"不进主链。

### C3 · 三级标注，诚实分层

每张卡右上角挂一个状态徽章，全站统一三级：

- **精做**：triage / threshold / fieldkit / mockhis / coder / settle / plugin-dashboard / ocr-import
- **预览**：trace / cases / data-health / 申诉批阅 / onsite 的 P1 技能
- **生产形态**：浏览器扩展（MV3）、桌面哨兵（剪贴板/文件夹）——**这两个是"装上就能用"这句话的实体证据，现在在页面上完全没位置**。plugins.html 底部加一条「生产形态」横条：扩展是真实 HIS 的接入方式（mockhis 页内标注"此为内嵌演示版，生产部署用浏览器扩展"）；桌面哨兵给出三通道对应场景（剪贴板=快速审 Excel、文件夹=现场散落材料、OCR=路线图）。

### C4 · 不合并的，写明为什么（防止过度收敛）

以下几组盘点时看着重复、实为不同阶段/不同口径，**保留并在页面上写清分工**：

- fieldkit（进场前 30 分钟的材料整理）vs onsite（进场后 3-7 天的执行闭环）——上下游，加衔接链即可（A2-10）。
- plugin-dashboard（院端自查口径）vs priority（监管转办口径）——同引擎两张表，数据同源、词汇必须不同（B4）。
- mockhis 浮层（演示形态）vs 浏览器扩展（生产形态）——同一脚本两种载体，页面标注即可（C3）。
- threshold 试算（管理员改参数）vs 主平台档位（用户选深度）——两个心智模型，不合并。
